## Context

The webhook pipeline today: `mountGitHubWebhook` verifies headers + HMAC and fire-and-forgets `handleGitHubWebhook` (A1, `events.ts:37-76`), which builds a `GitHubEventContext` and calls `routeGitHubEvent` (`webhooks.ts:433`). The router's decision flow (A5): build context → self-event filter (`context.user === '{slug}[bot]'`, machine events exempt, `webhooks.ts:444-451`) → branch→taskId extraction → `findTaskByPRNumber` fallback for `issue_comment`/`check_suite` (`webhooks.ts:458-466`) → **discard "Not our branch pattern"** when no taskId (`webhooks.ts:470`) → `determineRouteAction` switch. `GitHubEventContext` has no `issueNumber` field and `formatGitHubContext` has no `issues` branch (A2, A3); `issues.opened` falls through with nothing extracted and dies at the no-taskId discard. The public route union is `discard` / `direct(merge_check | existing_task | checks_ready)` (A4, `webhooks.ts:346-355`).

Task creation has exactly three call sites, all following create → link channel → seed → `sendMessage(newTask)` (B1, B2); the trigger scheduler is the precedent for non-Slack creation (B3). `ChannelType` already includes `'github'` and a PR-shaped `GitHubChannel` type exists **but is never constructed anywhere** (B5, J3); its two read-side consumers are `spawn.ts:309` (PM context line) and `memory/lifecycle.ts:377-383` (links block). `postToUser` handles `slack` and `cli`; a github-typed default channel silently falls to `return null` (B6, `task.ts:447-460`). Comment dedup exists only on PR branch states (A9); plain-issue threads have no dedup store. The GitHub client has `addPRComment` — already the issues endpoint, so it posts to plain issues (C1) — but no reaction or permission helpers (C2). Since the dossier was cut, `main` gained the merge/max-mode/trigger approval machinery (pr-merge-policy): `postInteractiveToUser`'s union is now `'edit_mode' | 'research_budget' | 'merge' | 'trigger' | 'max_mode'` and `handleMergeApproval` exists with a disposition-return pattern (`task.ts:1270-1273`) this design reuses for the edit-approval guard.

## Goals / Non-Goals

**Goals** (brief): mention → new task seeded with full context; the GitHub thread is the conversation surface (ack in place, PM replies as comments); write/maintain/admin authors only — for the summoning mention *and* for follow-up comments routed into the task; follow-ups route to the same task deduplicated; polite decline in uncovered repos; readonly always, enforced by construction.

**Non-Goals** (brief): no edit mode or approval flow for GitHub-born tasks; no Slack presence; no `pull_request_review_comment` mentions; no `@archie` alias or machine user; no behavior change for mentions resolving to existing tasks; mentions edited into existing comments or issue bodies do not trigger — `issue_comment.edited` and `issues.edited` stay unrouted, matching today's `action !== 'created'` noop and claude-code-action's created-only default (H6); and — constraints — no change to `merge_check`/`existing_task`/`checks_ready` routing, no new auth infrastructure, no PM-message silent drops, no silent `request_edit_mode` hang.

## Decisions

### 1. Router: intercept at the no-taskId discard with a new `new_task` route variant

`GitHubRouteResult` gains `| { action: 'direct'; handler: 'new_task'; mention: GitHubMentionEvent }` (A4). The interception point is exactly the `if (!taskId)` discard (A5, `webhooks.ts:469-471`): resolution attempts stay first (branch pattern → `findTaskByPRNumber` → new issue→task mapping, Decision 8), so a mention that resolves to an existing task keeps today's `existing_task` behavior by construction (brief non-goal 5), and the self-event filter has already run (loop safety, Decision 9). When no task resolves and the event is mention-eligible with a matching body, return `new_task`; otherwise the discard reason stays byte-identical. `determineRouteAction` is **not modified** — follow-up comments resolved via the issue mapping flow through its existing `issue_comment created → existing_task` arm (`webhooks.ts:378-380`), and a redelivered `issues.opened` for an already-mapped issue resolves via the mapping — which Decision 8 wires for `issues` events too, precisely so redelivery cannot re-enter the mention path and mint a duplicate task — and falls to the existing `default → noop` discard. Dispatch (`events.ts:131-139`) gains one case calling the new sibling handler `handleGitHubMentionDirect` (A7).

`GitHubMentionEvent` carries what the handler and AC1 need: `githubRepo`, `issueNumber`, `isPr`, `author` (= `context.user`, the same sender identity the self-filter vetted), `commentId?` + `commentBody?` (comment-born only), `issueTitle`, `issueBody?`, `issueAuthor?`, `htmlUrl` (comment or issue URL). Extracted from the raw payload in the router — the lean `GitHubEventContext` is not the vehicle for one-shot seeding data.

### 2. Context extraction: `issueNumber` for all `issue_comment` events plus an `issues` branch

`GitHubEventContext` gains `issueNumber?: number`. The `issue_comment` branch of `formatGitHubContext` (`webhooks.ts:97-104`) sets `issueNumber = issue.number` and `body`/`commentId` for **all** issue comments, keeping `prNumber` set only when `issue.pull_request` is present (H1) — nothing in the existing paths reads the new field, so AC11 is unaffected. A new `issues` branch (A3) sets `issueNumber`, `body` (issue body), and `state`. `formatGitHubEvent`'s `issue_comment` case renders destination `issue #N` when only `issueNumber` is set, keeping `PR #N` whenever `prNumber` exists — Archie-managed PR comments render byte-identically (AC11, pinned by an explicit `formatGitHubEvent` regression assertion).

### 3. Mention detection: word-boundary regex on the app slug, inert without the slug, bot authors excluded

`matchesMention(body, slug)` uses claude-code-action's proven matcher (H6): case-insensitive `(^|\s)@${escapeRegExp(slug)}([\s.,!?;:]|$)` (small local `escapeRegExp` — the codebase has none). Eligible events: `issue_comment` with `action === 'created'` (comment body) and `issues` with `action === 'opened'` (issue body) — exactly the brief's two surfaces; `edited` actions are never eligible (non-goals), and detection reads **bodies only** — a mention appearing only in an issue *title* does not trigger. A mention inside a fenced code block **does** match — the regex has no markdown awareness; accepted as claude-code-action parity (H6) rather than adding a markdown parser. The slug comes from `GITHUB_APP_SLUG` via the existing `getGitHubAppBotUsername` source (A6); when unset, detection returns no match — the mention feature is **inert**, never a fallback match (the slug is load-bearing twice: detection target and self-filter, A6). Authors ending in `[bot]` are skipped even when they aren't our bot — claude-code-action parity (H6 rejects non-User actors) and bot-to-bot loop hygiene beyond our own self-filter (F3); collaborator-permission lookups for apps would fail closed anyway, so this only short-circuits noise before an API call.

### 4. Authorization gate: legacy `permission ∈ {admin, write}`, fail closed, silent discard

New client helper `getCollaboratorPermission(githubRepo, username)` wrapping `GET /repos/{owner}/{repo}/collaborators/{username}/permission` on the installation Octokit (C3 — no new auth infrastructure), returning the legacy `permission` field where maintain→write and triage→read (H4), so accepting `admin | write` implements the brief's "write/maintain/admin" exactly. The check lives in `handleGitHubMentionDirect` (not the router): the router stays a pure payload→decision function unit-testable without a GitHub client (AC1), and the handler owns all side-effectful gates (AC2/AC3). Unauthorized (`read`/`none`) → log the reason, create nothing, post nothing (AC3 — silent, no probing surface per the signed decision). A permission-API error or a null `getGitHubClient()` (C3 warns-and-nulls on missing config) also discards with a logged reason — **fail closed**; a transient GitHub blip costs the summoner a retry, never an unauthorized task. The same gate is re-applied to follow-up comments on GitHub-born threads (Decision 10).

### 5. Repo coverage gate: `findAgentDefsContainingRepo`, polite decline after authorization

After the permission gate: `findAgentDefsContainingRepo(githubRepo).length === 0` (D3 — exists with production callers) → post a polite decline via `addPRComment` and stop (AC9). Decline ordering matters: it runs after authorization so declines are reserved for authorized users (brief design decision — no probing surface for strangers). Decline text is friendly and mention-free (Decision 9), e.g. "Thanks for the mention! I don't currently cover this repository — ask an Archie admin to add it to my plugin configuration." Because no task and no dedup store exist on this path, repeated mentions would each post a decline; a small in-memory dedup — `Map<'repo#issue', lastDeclinedAt>` with a ~10-minute window, entries older than the window evicted lazily on access so the map cannot grow unboundedly across threads — caps an authorized user's burst at one comment per thread per window (process-local and lost on restart, which is fine: the cost of a duplicate decline after a restart is one comment). No broader task-creation cap is added; the trigger path's `DAILY_FIRE_CAP` (A10) is the precedent if one is ever wanted. Known trade-off: undeclared repos are technically reachable at runtime via PM-spawned dynamic agents + lazy clone (D4, J6), so this is a **product gate as signed**, not a technical necessity — v2 could route uncovered-repo mentions to a PM armed with `spawn_repo_agent` instead.

### 6. Task creation: Slack-sequence analog; the GitHub channel entry IS the origin record

`handleGitHubMentionDirect` mirrors the Slack after-create sequence (B2: create → link → ack → prompt): `Task.create()` → `task.linkGitHubChannel(repo, issueNumber, isPr)` (Decision 7) → **`await task.save(true)`** → seed knowledge.log (Decision 8) → `metadata.title ??= issueTitle` (the issue title already is a one-line summary; no LLM title pass needed) → acknowledge → `task.sendMessage(AGENT_PROMPTS.newTask, 'pm-agent')` (B4 — the seeded log carries the context, so the standard prompt suffices; the trigger-style dedicated prompt (B3) is unnecessary).

**No separate `github_origin` metadata field.** The linked GitHub channel entry already records the origin — repo, issue number, PR flag — and in v1 nothing else ever constructs a github channel, so "task has a github channel" is exactly "task is GitHub-born". A new `Task.isGitHubBorn()` helper (any `channels` value with `type === 'github'`) is the single predicate the readonly guards (Decision 11) and PM context (Decision 12) read. This drops a metadata field, a seeding step, and a second `"issue_number":` substring hit from the mapping scan (Decision 8's structural verification would tolerate the noise, but why create it). The summoner's login is not lost — it is the `from` of the seeded knowledge-log entries (Decision 8). Trade-off recorded below: if a future version links github channels to non-GitHub-born tasks (e.g. Slack cross-post, a v2 non-goal today), an explicit origin field must return; the helper centralizes the predicate so that swap is one function body.

**Durability**: the channel entry doubles as the readonly marker, so it must not sit in `debouncedSave`'s 500ms window — a crash inside the window would leave a task directory whose on-disk metadata lacks the marker, reopening the approve-API hole (E4) for that task. The handler persists synchronously via `save(true)` right after linking, the same flush-before-it-matters reasoning `handleEditModeApproval` applies to `edit_allowed` (`task.ts:1218-1221`).

Acknowledgment (AC5): an `eyes` reaction on the triggering comment — or on the issue itself for `issues.opened` — plus one short comment naming the task id (e.g. "On it — created `task-…`. I'll follow up here."). Reactions need two net-new `GitHubClient` methods (C2 — none exist): `addCommentReaction(githubRepo, commentId, content = 'eyes')` → `POST /repos/{o}/{r}/issues/comments/{comment_id}/reactions` and `addIssueReaction(githubRepo, issueNumber, content = 'eyes')` → `POST /repos/{o}/{r}/issues/{issue_number}/reactions` (H3; both covered by Issues: write, octokit `reactions.createForIssueComment`/`createForIssue` semantics via the existing `request` style). The naming comment reuses `addPRComment` unchanged — it already posts to the issues endpoint, so it works for plain issues despite the name (C1, J2). Ack failures log a warn and never abort creation: the reaction is UX, the task is the product.

Failure after authorization: if creation/linking/seeding throws after the permission gate passed, the webhook wrapper's try/catch (`events.ts:66-73`) logs the error and the summoner sees nothing — no ack, no task activity. Accepted for v1: the webhook is fire-and-forget by design (A1), the failure is operator-visible in logs, and the user's remedy is to re-mention; a partially created task directory without a sent PM message stays inert.

Race note: two near-simultaneous mentions on the same not-yet-mapped issue can both route `new_task` (the mapping is a lockless fs scan, B8). The handler re-checks `findTaskByIssueChannel` after the permission gate and falls through to the existing-task delivery path when a task appeared meanwhile — narrowing, not closing, the window (single-Task-instance work is issue #201; same class as today's `findTaskByPRNumber`). Accepted (Known trade-offs).

### 7. GitHub channel: reshape the never-constructed type, `linkGitHubChannel` analog of `linkCliChannel`

`GitHubChannel` (B5, `types/task.ts:121-125`) is redefined — safe because nothing constructs it, so zero on-disk instances exist (J3):

```typescript
export interface GitHubChannel extends ChannelBase {
  type: 'github';
  repo: string;
  issue_number: number;   // issue or PR number — GitHub's issues endpoints address both (C1/H1)
  is_pr: boolean;         // rendering + link building only
  last_processed_comment_id?: number;  // follow-up dedup for this thread (Decision 8/10)
}
```

`Task.linkGitHubChannel(repo, issueNumber, isPr)` mirrors `linkCliChannel` (B2, `task.ts:411-415`): write `channels['github:{repo}#{issueNumber}']`, promote `default_channel ??=`, save; idempotent on re-link (persistence timing per Decision 6). The two read-side consumers update: `spawn.ts:309` renders `GitHub {repo}#{n} (PR)` / `(issue)`, and `memory/lifecycle.ts:377-383` builds `/pull/{n}` vs `/issues/{n}` URLs from `is_pr` — **removing the `(channel as { pr_number?: number }).pr_number` cast at `lifecycle.ts:378-379`**, which would otherwise make the reshape a silent `undefined` (dropped link) instead of a compile error; the `else if` already narrows the type, so plain property access is both safer and simpler.

`postToUser` (B6) gains a `github` branch in **both** arms — the default-channel arm (AC6's target) and the explicit `target.channel` arm (the PM prompt documents channel targeting; a github key must not silently no-op): post via `addPRComment(ch.repo, ch.issue_number, message + footer)` with the standard footer (`task.ts:434`) appended as a trailing italicized line (GitHub has no context blocks), then `logOutgoingMessage(sender, message, 'github:{repo}#{n}')`. The `addPRComment` call is wrapped in try/catch — a locked/closed/transferred issue or a rate-limit rejection logs a warn and continues rather than propagating into the PM's tool call (Slack's `postSlackMessage` failures don't crash the PM either; parity). A null client logs a warn — never the silent `return null`. Adjacent hardening: `postFilesToUser` warns "file upload to GitHub channels is not supported — files dropped" instead of today's silent fall-through (same silent-drop class the brief's constraint targets). Everything else stays B7's graceful degradation: `resolveSlackChannel` returns null so reactions no-op with a warn and `postInteractiveToUser` falls back to the log line — acceptable because Decision 11 makes the approval-prompt paths unreachable for the tools that matter.

### 8. Knowledge-log seeding and the issue→task mapping

Seeding uses `appendGitHubEvent` verbatim-message semantics (A8, B9). Comment-born tasks get **two** entries: (1) thread context — `from: issueAuthor ?? author`, `destination: 'issue #N' | 'PR #N'`, message `opened "{title}"\n\n{issue body}\n\n{issue html_url}`; (2) the mentioning comment in the exact existing `issue_comment` shape — `{comment body} [comment_id={id}]` (matching `webhooks.ts:226-227`) plus the comment's `html_url` — so AC2's full list (repo + number via the `github:{repo}/{destination}` prefix `persistence.ts:482`, title, body, mentioning text, author, link back) lands in the log. Issue-born tasks (`issues.opened`) get one entry — the body is the mentioning text. At creation the channel's `last_processed_comment_id` is set to the triggering `commentId` (comment-born only): the seed consumed that comment, so a webhook redelivery — which now resolves via the mapping to `existing_task` — dedups to a no-op.

`findTaskByIssueChannel(githubRepo, issueNumber)` in `persistence.ts` follows the established pattern (B8): check in-memory `activeTasks` for a matching github channel first (like `findTaskByThread`, `persistence.ts:559-573`), then `scanMetadataFiles('"issue_number": N')` and structurally verify a `channels` entry with `type === 'github'`, matching repo and number. Wired into `routeGitHubEvent` as a resolution step for **both** `issue_comment` events (third step, after the PR-number fallback `webhooks.ts:458-460` — ordering per AC1: branch pattern → PR lookup → issue mapping) **and** `issues` events (so a redelivered `issues.opened` resolves to its task and discards via the `noop` arm instead of re-entering the mention path and creating a duplicate task + duplicate ack, Decision 1). The mapping consult is **slug-gated** (Decision 9): when `GITHUB_APP_SLUG` is unset it is skipped entirely, making the whole GitHub-born surface — detection *and* follow-up routing — inert together.

Disk-I/O note: this consult adds a full `SESSIONS_DIR` metadata scan to every `issue_comment` that resolves no task — previously a plain-issue comment (no `prNumber`, so no `findTaskByPRNumber` fallback) discarded scan-free; now it reads every task's `metadata.json` once before the same discard. The in-memory `activeTasks` short-circuit only saves the scan for live tasks; misses pay full price — the identical cost profile `findTaskByPRNumber` already has on every PR-comment miss (B8). Bounded by session-directory count and fine at expected webhook volume; a real index is issue #201 territory (task indexing / single-Task-instance), no v1 change beyond this note.

### 9. Loop safety: self-filter first, mention-free bot text, slug-gated GitHub-born routing

Mention detection runs strictly after the self-event filter (brief constraint; A5 placement), so Archie's own acks and PM comments — authored `{slug}[bot]` (H5) — discard as `Own bot event` before detection (AC8), and detection itself requires the slug (Decision 3), so the mention path cannot run while the self-filter is disabled (A6). Defense in depth: the ack, decline, and PM-comment templates never contain the mention string (today no Archie-authored GitHub text can contain it, F3 — this change keeps that invariant), and non-our-bot `[bot]` authors are skipped (Decision 3).

The dangerous residual was follow-ups, not detection: a task created while the slug was set keeps routing follow-up comments via the issue mapping even after the slug is unset — with the self-filter then off, a PM comment would re-wake its own task and the PM could reply again, an **unbounded** loop capped only by GitHub's secondary limits (H7). Structural fix rather than a warning: the issue-mapping consult itself is gated on the slug being set (Decision 8), exactly like detection — with the slug unset, GitHub-born threads stop routing at all (follow-ups are dropped along with new mentions; the feature is uniformly inert), so the loop's inbound leg cannot exist. The Archie-managed-PR path is unaffected by the gate (it resolves via branch/PR lookups, not the mapping), preserving today's behavior with an unset slug (AC11). The boot warning stays for visibility: `src/index.ts` warns when `GITHUB_WEBHOOK_SECRET` is set but `GITHUB_APP_SLUG` is not ("self-event filter, mention trigger, and GitHub-born task routing disabled"), following the conditional-validation precedent (F6, `validateMasterKey` `index.ts:102-105`) — a warn, not a boot failure, since GitHub config is lazy and feature-gating by design (C3, F6).

### 10. Follow-up comments: permission-gated existing-task delivery with channel-level dedup

Follow-ups resolve via Decision 8's mapping and flow into `handleExistingTaskDirect` (A8, `events.ts:267-304`).

**Author gate (required)**: without one, this feature would open an ingress that does not exist today — currently untrusted comments on non-Archie threads die at the router, and readonly bounds only *writes*: a readonly PM can still READ any installation-accessible repo (dynamic repo agents + lazy clone, D4) and relay content onto the thread via `postToUser`, so on a public repo any GitHub account could post a follow-up steering the PM to exfiltrate private-repo content onto the public issue. Therefore: when the event is an `issue_comment` on a task whose matching thread is a github channel (i.e. the GitHub-born path), `handleExistingTaskDirect` re-checks the author's permission before anything else and **silently drops** `read`/`none` authors (logged reason, no reply, no knowledge-log append, no PM wake, and no watermark advance — a dropped comment id stays unconsumed, so behavior is stable across redeliveries). Fail closed on lookup errors, same as Decision 4. No mention is required in follow-ups — authorized users converse naturally (AC7 UX). Two hardenings keep the gate itself from becoming a lever: (a) `[bot]`-suffixed authors are short-circuited **before** any permission GET — Decision 3 already does this for detection, and the follow-up gate mirrors it (our own bot never reaches here, the self-filter runs upstream); (b) permission results are **cached in v1** in a small in-memory per-`(repo, author)` map with a short TTL (~5 minutes, lazily evicted on access like Decision 5's map), caching resolved outcomes including `read`/`none` — negative caching is the point: every ungated GET spends the shared installation token's 5,000 req/h **primary** budget (H7), so sustained multi-account comment spam on a public thread would otherwise degrade ALL of Archie's GitHub operations (merge checks, checks-ready, PR-card refreshes), not just this feature. Thrown lookups are not cached (a transient blip shouldn't pin an author to fail-closed for the TTL). Staleness cost: a revocation takes effect within the TTL — an author whose write access was just removed can steer an existing task for up to ~5 more minutes; accepted (Known trade-offs). The Archie-managed-PR comment path is **not** gated — it resolves via branch/PR lookup, not the mapping, and changing it would violate AC11 (its ungated-ingress status quo predates this change and is out of scope).

**v2 caveat — the gate's coverage is shaped by readonly-v1**: its safety currently rests on two facts — GitHub-born tasks cannot write (so the worst injection outcome is read-and-relay, which the gate blocks), and the gate keys on the exact `repo#number` channel match. If v2 relaxes readonly, an Archie-opened PR from a GitHub-born task gets a **different** issue number: comments there resolve via `findTaskByPRNumber`/branch pattern and would bypass this gate entirely, reopening B1 with write access in play. When readonly is relaxed, the gate must either key on `isGitHubBorn()` (any github channel on the resolved task, regardless of which thread matched) or Archie-opened PRs must be linked as additional github channels so the per-thread match still covers them. Recorded here so v2 does not rediscover this the hard way.

**Dedup**: today's dedup is PR-branch-state-scoped (A9) and its guard condition requires `context.prNumber` — plain-issue comments would bypass dedup entirely. Extension: for authorized comments where `context.commentId` is set and the task has a github channel matching `(repo, issueNumber ?? prNumber)`, fold the channel's `last_processed_comment_id` into the same max/skip/advance computation and advance it alongside any branch-state matches (`task.debouncedSave()` as today). For Archie-managed PR tasks there is no github channel entry, so the computation reduces to today's exact behavior — pinned by regression tests on the refactored handler (AC11); for GitHub-born tasks (plain issues *and* non-Archie PRs — which have no branch states) the channel entry is the dedup store. Delivery is unchanged: `appendGitHubEvent` + `sendMessage(existingTask, 'pm-agent')` (AC7) — the PM's reply then lands on the thread via Decision 7.

### 11. Readonly v1: fail-fast tool guards plus an origin guard in the approval choke point

All guards read the single predicate `task.isGitHubBorn()` (Decision 6). Three layers close E2-E5/J5:

- **`request_edit_mode`** (`tools.ts:596`): first check in the handler — GitHub-born → return an ok-shaped explanation ("GitHub-born tasks are read-only in v1 — edit mode cannot be requested or granted. Tell the user that code changes require starting the request from Slack.") — no interactive post, no `suspendStatus`, no `deferTeardown`, no pause (AC10; today it would post to a log line and park the task forever, E2).
- **`request_max_mode`** (`tools.ts:679`): identical defer-stop pattern (E3), identical silent-hang failure mode → same-shaped guard ("max mode has no approval surface on a GitHub-born task in v1 — continue in standard mode"). The third defer-stop site, `merge_pull_request`'s approval request (`tools.ts:1793-1814`), needs no guard: the tool is read-only-disallowed (`spawn.ts:506`) and edit mode is unreachable on these tasks, so it is unreachable by construction (E1). The fourth `postInteractiveToUser` at `tools.ts:2327` (trigger proposal) does not defer-stop (E3) and is left alone. The research-budget pause (`task.ts:1184-1194`) remains reachable — see Known trade-offs.
- **`handleEditModeApproval`** (`task.ts:1199`): first line — GitHub-born → log a warn, append a decision finding ("Edit mode approval rejected — GitHub-born tasks are read-only in v1"), and return a `'rejected_readonly'` disposition **without** setting `edit_allowed`, recording an approver, saving, or restarting agents (E5's unconditional flip is what this guards). The return type widens from `void` to a disposition, mirroring `handleMergeApproval`'s `'resolved' | 'stale'` pattern. The API route (`routes.ts:274-279`) maps the rejection to `403 { error: 'GitHub-born tasks are read-only in v1' }` and skips the `approval:resolved` emit — closing the unauthenticated approve-API hole (E4: no auth middleware, binds all interfaces) for GitHub-born tasks specifically; the Slack `approve_edit_mode` handler tolerates the disposition (unreachable anyway — these tasks have no Slack channel, and no prompt is ever posted). Since spawn-time gating reads only `edit_allowed` (E1, `spawn.ts:381`), never-set means readonly-by-construction. The marker the predicate reads is persisted synchronously at creation (Decision 6), so a crash cannot produce a GitHub-born task the guard doesn't recognize.

### 12. PM prompt and spawn context for GitHub-born tasks

The PM prompt is Slack-centric (F4 — 31 matches). Two adjustments, minimal by design: (1) a dynamic context line in `spawn.ts`'s PM `contextLines` (the `triggered_by` precedent, `spawn.ts:333-334`) when the task is GitHub-born (Decision 6 predicate) — origin repo/number/URL from the github channel entry, "replies via `post_to_user` land as comments on that GitHub thread; there is no Slack thread", and "this task is read-only for its lifetime (v1): never call `request_edit_mode` or `request_max_mode`; if asked for code changes, explain the request must start from Slack"; (2) a short "GitHub-born tasks" subsection in `prompts/pm-agent.md` stating the same rules generally plus "Slack-specific tools (`react_to_message`, `find_slack_user`, threads) do not apply". The renderChannel update (Decision 7) makes the channel list itself say `GitHub {repo}#{n}`.

No new env vars; no boot-requirement change beyond Decision 9's warning (F6). ### 14. Testing approach

## Known trade-offs

- **`postToUser` gains a `github` branch that issue #203 wants to delete** (I3): #203 reworks `postToUser` into engine-emitted semantic events with channel subscribers — this change deepens the coupling it would remove. Accepted for v1: the branch is ~10 lines, isolated behind the channel type switch, and migrates naturally into a subscriber if #203 lands.
- **Uncovered-repo decline is a product gate, not a technical one** (D4, J6): dynamic repo agents + lazy clone can already serve any installation-accessible repo. Shipping the conservative decline as signed; the v2 alternative (route to a PM with `spawn_repo_agent`) is one handler-branch swap away.
- **Fail-closed permission checks drop authorized traffic during GitHub API blips**: a summoner sees no reaction; an authorized follow-up is silently ignored. The author retries or re-comments. Preferred over any fail-open posture on an unauthenticated ingress.
- **Follow-up permission GETs spend the shared installation primary budget** (Decision 10, H7): ungated, multi-account comment spam by *read-permission* accounts on a public thread could burn the 5,000 req/h token budget shared by every Archie GitHub operation (merge checks, checks-ready, card refreshes) — a cheap DoS needing no write access. Bounded in v1 by the per-`(repo, author)` short-TTL cache (negative results cached too) and the pre-GET `[bot]` short-circuit; residual exposure is one GET per distinct spamming account per TTL window, accepted. The cache's flip side: permission revocation propagates within the TTL (~5 min), so a just-revoked author can steer an existing task briefly — accepted. No rate cap on mention-born task creation either (A10) — the permission gate is the control, GitHub's secondary limits bound any runaway loop, the decline dedup (Decision 5) caps decline spam, and `DAILY_FIRE_CAP` is the precedent if a broader cap is ever wanted. A mention storm from a write-permission user is a social problem v1 accepts.
- **Prompt-injection surface widens to anyone with repo write — and no further** (F1): both the summoning mention (Decision 4) and every routed follow-up (Decision 10) are permission-gated, so only write/maintain/admin authors' text reaches the PM; bodies flow verbatim into knowledge.log exactly as authorized PR comments already do today (A8, B9). The trust boundary doc's "authenticate-then-trust" model is matched, not exceeded. No escaping layer exists to reuse (F2) and none is introduced.
- **The follow-up gate's coverage is load-bearing on readonly-v1 + the exact channel match** (Decision 10's v2 caveat): comments on an Archie-opened PR (different number, resolved via `findTaskByPRNumber`) would bypass the gate — harmless while GitHub-born tasks cannot write, but any v2 readonly relaxation must first re-key the gate on `isGitHubBorn()` or link Archie-opened PRs as additional github channels, or B1 reopens with writes enabled.
- **Readonly-born detection is derived, not declared** (Decision 6): `isGitHubBorn()` reads "has a github channel", which is exact in v1 (only the mention handler constructs them) but becomes wrong if a future feature links github channels to tasks born elsewhere — at which point an explicit origin field must return; the single-predicate helper makes that a one-function swap. Chosen over a redundant `github_origin` metadata field (less state, no double bookkeeping, no extra scan noise).
- **Research-budget approval on a GitHub-born task has no in-thread surface**: `web_research` budget exhaustion still posts its prompt to the log-line fallback and parks the task (E2 nuance); the `approval:requested` event remains resolvable via API/CLI/debug-MCP (E2), so it is stuck-but-recoverable, not silent-and-lost. Guarding it is out of the brief's scope; noted for the #203-era rework.
- **The unauthenticated approve API can still flip `max_mode` or research budget on a GitHub-born task** (E4): cost/capability knobs, not write access — readonly-by-construction concerns `edit_allowed` only (E5). The API-auth gap itself is pre-existing and out of scope.
- **Issue→task mapping races are narrowed, not closed** (Decision 6): lockless fs-scan mapping matches the existing `findTaskByPRNumber` semantics; the real fix is issue #201's single-Task-instance work.
- **Creation failure after authorization is silent to the summoner** (Decision 6): fire-and-forget webhook semantics (A1); logged for operators, re-mention is the remedy.

## What this change does NOT touch (AC11)

`determineRouteAction`'s existing arms and their outputs — `merge_check`, `checks_ready`, and every `existing_task` decision for PR-attributed events — are byte-for-byte untouched, as are `handleMergeCheckDirect`, `handleChecksReadyDirect`, `merge.ts`, branch-state dedup for Archie-managed PRs, and the PR-card refresh path. The follow-up author gate (Decision 10) applies only to threads resolved via the issue→task mapping (GitHub-born); the Archie-managed PR comment path — including its existing ungated ingress — is unchanged. The self-event filter condition (`webhooks.ts:444-451`) is unchanged — new code runs only downstream of it. No `pull_request_review_comment` handling is added. No Slack code paths change.

## Migration / Rollback

No data migration: the reshaped `GitHubChannel` is never-persisted at HEAD (J3) and appears only on newly created GitHub-born tasks. Rollout: deploy archie-hq, then AC13's post-merge step (GitHub App settings: subscribe `issues`, keep `issue_comment`, confirm Issues: read & write permission — reactions and plain-issue comments need Issues: write, H2/H3). Rollback: revert the deploy — old code discards `issues` events at the no-taskId discard and tolerates unknown channel entries in metadata, so leftover App settings and GitHub-born task dirs are harmless. Caveat: in-flight GitHub-born tasks lose their delivery surface on rollback (old `postToUser` has no github branch → messages drop with the no-default-channel warn at best); operator remedy is to let them idle out or stop them — they are readonly, so nothing is half-written.
