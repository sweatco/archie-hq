# Research dossier — `github-mention-trigger`

Ground truth for planning. All `file:line` citations verified against the run branch `forge/github-mention-trigger` rebased onto `origin/main` @ `3ed6ba4` (2026-07-11) by the adversarial verification pass (`verdicts/research-verifier-round1.md`). Only CONFIRMED claims remain.

## A. Webhook ingress and routing (current state)

- A1. Webhook mount: `mountGitHubWebhook` registers `POST /webhooks/github` with `express.raw`, checks headers (400, `events.ts:51-54`), verifies HMAC-SHA256 with `timingSafeEqual` (401, `events.ts:57-61`; `verifyWebhookSignature` `src/connectors/github/webhooks.ts:27-35`), acks 200 immediately (`events.ts:64`), then fire-and-forget async handling (`events.ts:66-73`). Mounted only when `GITHUB_WEBHOOK_SECRET` set (`src/index.ts:231-233`).
- A2. `GitHubEventContext` has `eventType, action, githubRepo, prNumber?, branch?, user, body?, state?, commentId?` — **no `issueNumber` field**; for `issue_comment`, `prNumber` is set only when `issue.pull_request` is present. `src/connectors/github/webhooks.ts:44-54, 97-104`.
- A3. `formatGitHubContext` has an `issue_comment` branch (`webhooks.ts:97`) but no `issues` branch (`webhooks.ts:59-141`) — `issues.opened` payloads produce no useful context today.
- A4. Route plumbing: `InternalRouteAction = 'merge_check' | 'existing_task' | 'checks_ready' | 'noop'` (`webhooks.ts:360`); `determineRouteAction` is a pure switch on `eventType` (`webhooks.ts:365`), `issue_comment created → existing_task` (`webhooks.ts:378-380`), `issues` falls to `default → noop` (`webhooks.ts:412-413`). Public `GitHubRouteResult` union: `discard` / `direct(merge_check|existing_task)` / `direct(checks_ready)` (`webhooks.ts:346-355`) — a `new_task` variant would be new.
- A5. `routeGitHubEvent` decision flow: build context → **self-event filter** (`context.user === '{GITHUB_APP_SLUG}[bot]'`, exempting machine events `check_suite`/`workflow_run`/`push`) at `webhooks.ts:444-451` → branch→taskId extraction → `findTaskByPRNumber` fallback for `issue_comment` (`webhooks.ts:458-460`) and `check_suite` (`webhooks.ts:464-466`) → **discard `'Not our branch pattern'`** when no taskId (`webhooks.ts:470`) → `determineRouteAction` + result switch. New-task mention handling must intercept at/before that discard, after the self-filter.
- A6. `getGitHubAppBotUsername()` returns **null when `GITHUB_APP_SLUG` unset** (`webhooks.ts:420-423`), and the filter is guarded `if (ourBotUsername && …)` (`webhooks.ts:449`) — **unset slug disables the self-event filter entirely**. The mention feature makes the slug load-bearing twice (detection target + loop safety).
- A7. Dispatch: `handleGitHubWebhook` switches on `route.handler` — `merge_check`/`existing_task`/`checks_ready` (`src/connectors/github/events.ts:131-139`). A `new_task` handler would be a new case + sibling function to `handleExistingTaskDirect` (`events.ts:267`).
- A8. Existing-task delivery pattern: `appendGitHubEvent(taskId, repo, formatGitHubEvent(context))` then `task.sendMessage(AGENT_PROMPTS.existingTask, 'pm-agent')` (`events.ts:302-303`); comment body lands verbatim in the knowledge log with `[comment_id=…]` tag (`formatGitHubEvent` `issue_comment` case `webhooks.ts:226-227`).
- A9. Comment dedup today is PR-branch-state-scoped only: max `last_processed_comment_id` across matching branch states, skip if `commentId <= lastProcessed`, else advance on all matches + `task.debouncedSave()` (`events.ts:273-300`). Keyed via `findBranchStateByPR` (`src/connectors/github/branch-state.ts:47`). `last_processed_comment_id` exists only on `BranchState` (`src/types/task.ts:170`) and legacy `RepositoryInfo` (`types/task.ts:240`) — **no issue-number analog**; plain-issue threads have no dedup store.
- A10. Webhook-path throttles: merge-check debounce 5s (`webhooks.ts:260-261`), checks-ready 20s (`webhooks.ts:298-299`), PR-card refresh 2.5s (`events.ts:149-150`). **No rate limiting on Slack/API/webhook task creation**; the one task-creation cap in the codebase is the trigger path's `DAILY_FIRE_CAP = 200` (`src/system/trigger-scheduler.ts:27`, enforced `:357-361`) — an existing precedent if mention-born tasks want a cap.

## B. Task creation, channels, PM delivery

- B1. `Task.create()` is the single creation path (`src/tasks/task.ts:184-221`); metadata starts `task_owner:null, participants:[], channels:{}, default_channel:null, repositories:{}` (`task.ts:200-211`). Exactly three call sites: Slack handler (`src/connectors/slack/events.ts:775`), API `POST /tasks` (`src/connectors/api/routes.ts:183` — also the CLI's path), trigger firing (`src/system/trigger-scheduler.ts:363`).
- B2. Slack after-create sequence: `Task.create()` → `task.append(thread)` (channel link, `slack/events.ts:776`) → ack (`:779-780`) → `task.sendMessage(AGENT_PROMPTS.newTask)` (`:787`). CLI/API analog: create → `task.linkCliChannel()` → `appendCliMessage(...)` → `sendMessage(newTask)` (`routes.ts:183-186`). `linkCliChannel` writes `channels[CLI_CHANNEL_KEY]` and promotes `default_channel ??=` (`task.ts:412-413`).
- B3. PR #206 (merged 2026-07-10) added **triggers**: `fireTrigger` (`trigger-scheduler.ts:336`) creates tasks via `Task.create()` (`:363`); for channel-message fires it links the Slack thread via `linkSlackThread(channelId, threadTs, channelName)` (`task.ts:887-902` — "posts nothing" = posts no Slack message); **schedule-fired tasks have no channel at all**; it appends nothing to knowledge.log and seeds the PM via `task.sendMessage(AGENT_PROMPTS.triggered(seed, reason), 'pm-agent')` (`:385`; prompt `src/agents/prompts.ts:26`). Closest precedent for non-Slack (even channel-less) task creation; PR #206's description names GitHub-event triggers as a deliberate follow-up.
- B4. `AGENT_PROMPTS.newTask = 'New task created, assign owner'`, `existingTask = 'New input received. Check knowledge.log for the update.'` (`src/agents/prompts.ts:10-11`).
- B5. Channel types: `ChannelType = 'slack' | 'github' | 'cli'` (`src/types/task.ts:89`); **`GitHubChannel` already exists but is PR-shaped and never constructed anywhere**: `{ type:'github'; repo:string; pr_number:number }` (`types/task.ts:121-125`), in the `Channel` union (`:135`). Two read-side consumers assume PR shape: knowledge-log links render `/pull/` URLs (`src/memory/lifecycle.ts:377-383`) and agent context renders `PR {repo}#{pr_number}` (`src/agents/spawn.ts:309`). Issues need an `issue_number` (or equivalent) extension.
- B6. `postToUser` branches: no default channel → warn `"…no default channel — message dropped"` (`task.ts:451`); `slack` (`:454`) and `cli` (`:457`) branches exist; **a `github`-type default channel matches neither and falls to `return null` (`:460`) with no warn at all**. AC6 requires a `github` branch here.
- B7. `resolveSlackChannel` returns the channel only if `type === 'slack'` (`task.ts:621`); `reactToMessage`/`unreactFromMessage` no-op with a warn (`task.ts:842, 856`), `ackMessage` returns fully silently (`:589-591`), `postInteractiveToUser` falls back to a log line (`:566-575`).
- B8. Task lookups are pure-fs scans over `SESSIONS_DIR/task-*/shared/metadata.json` (`scanMetadataFiles`, `src/tasks/persistence.ts:535-553`): `findTaskByPRNumber` substring-matches `'"pr_number": N'` then verifies via branch states (`persistence.ts:582-619`); `findTaskByThread` checks in-memory active tasks first, then scans (`:559-573`). An issue→task mapping would follow the same scan pattern over a new metadata key (branch states can't represent non-PR issues).
- B9. Knowledge-log append for GitHub events: `appendGitHubEvent` writes `event.message` unmodified (`src/tasks/persistence.ts:477-496`).

## E. Readonly / edit mode

- E1. Readonly enforcement is spawn-time: `edit_allowed === true` gates write mode (`src/agents/spawn.ts:381`); RO spawns get write-side MCP tools disallowed — incl. `push_branch` (`:498`), `create_pull_request` (`:499`), `add_pr_comment` (`:501`) — full list `spawn.ts:493-510`; clone paths are excluded from `allowWritePaths` and put in `denyWritePaths` (`:514-525`).
- E2. `request_edit_mode` tool (`src/agents/tools.ts:596`): appends a finding, builds Slack blocks, `task.postInteractiveToUser(..., 'edit_mode', args.channel)` (`:664`), then `agent.deferTeardown(() => task.stop())` (`:671`). On a task whose default channel is not Slack, `postInteractiveToUser` falls back to a log line (`task.ts:573-575`) but **the task still pauses**. Nuance: it also emits `approval:requested` on the event bus (`task.ts:559-564`), so SSE/CLI observers could still see and approve via API — the silent-hang holds for GitHub-born tasks with no CLI attached (AC10).
- E3. **`request_max_mode` follows the identical pattern** (`tools.ts:679, 747, 754`) and a third defer-stop site exists (`tools.ts:1793-1814`) — the AC10 guard must consider all defer-stop request tools, not just edit mode. (A fourth `postInteractiveToUser` at `:2327` does not defer-stop.)
- E4. ⚠ **Tension with "readonly always v1"**: the `/api` router has **no auth middleware** (only `router.use(express.json())`, `src/connectors/api/routes.ts:35`); `POST /tasks/:id/approve` (`routes.ts:221-328`) with `{type:'edit_mode', approve:true}` calls `task.handleEditModeApproval(cleanApprover)` (`:274-276`) for **any task, no origin/participant check**; mounted `app.use('/api', router)` (`:450`); server binds all interfaces (`server.listen(config.port, resolve)`, no host arg, `src/index.ts:264`). Blocking `request_edit_mode` alone does not make GitHub-born tasks readonly-by-construction.
- E5. `handleEditModeApproval(approver?)` (`src/tasks/task.ts:1199-1243`): clears pending PM teardown (`:1208`), sets `edit_allowed = true` **unconditionally** (`:1209`), records `edit_approved_by` first-wins when an approver is passed (`:1215-1217`), synchronous save (`:1221`), restarts running repo agents for writable mounts (`:1231-1238`), appends approval finding (`:1240-1241`). An approver-less POST still flips `edit_allowed`. **No origin or participant condition guards any of it.**
- E6. PRs #198/#191 (merged 2026-07-08) added the restart-on-approval mechanics (`task.ts:1223-1238`) — recent, adjacent to any AC10 change.

## F. Security model and conventions

- F1. Trust boundary doc: "Only web content is untrusted" (`docs/architecture/security.md:23`, section from `:21`) — Slack/GitHub content from authenticated senders is trusted verbatim; prompt-injection defenses (guardrails, defense tags) wrap only the `web_research` path (`security.md:126-162`); content-level injection detection for non-web content is explicitly not implemented (`security.md:336-341`). GitHub comment bodies already flow verbatim into knowledge logs today (A8, B9). The feature widens who can inject text to "anyone with repo write" — matching, not exceeding, current precedent; the permission gate (AC3) is the control.
- F2. Slack precedent for inbound text is authenticate-then-trust-verbatim; the only transformation is redaction of external-team participants in shared channels (`'[redacted: external participant in shared channel]'`, `src/tasks/persistence.ts:246-247`, rendering `:235-310`; `isExternalUser` gating in `src/connectors/slack/events.ts:259, 555, 674, 874, 957, 978`). No escaping layer exists to reuse.
- F3. RO agents cannot post GitHub comments via tools (`add_pr_comment` RO-disallowed, `spawn.ts:501`) and merge/CI flows post no comments (`src/connectors/github/merge.ts`: zero comment-posting) — today no Archie-authored GitHub text can contain the mention string. The feature's new ack + PM comments are bot-authored and will hit the self-filter (A5) — provided `GITHUB_APP_SLUG` is set (A6).
- F4. PM system prompt is Slack-centric (31 case-insensitive matches in `prompts/pm-agent.md`): opening framing (`:1`), `find_slack_user` (`:119`), `react_to_message` (`:121-122`), "New task from Slack" (`:336`), "Acknowledge in Slack" (`:339`). GitHub-born tasks need prompt/context adjustments or the PM will act on wrong assumptions.
- F5. Conventions: unified logger mandatory (CLAUDE.md); vitest runner (`package.json:25`); `"lint": "eslint src/**/*.ts"` (`package.json:13`) but **no ESLint config file exists in-repo** (may fail/no-op — don't rely on lint).
- F6. Boot validation: only `ANTHROPIC_API_KEY` hard-required (`src/index.ts:68-70`); GitHub config is lazy and feature-gating (A1, C3). Precedent for conditional validation: `validateMasterKey()` (`src/index.ts:102-105`).

## H. External facts (GitHub platform, claude-code-action)

- H1. `issue_comment` fires for comments on **both issues and PRs**; PRs are identified by the presence of `issue.pull_request` in the payload (the code already keys off this, `webhooks.ts:100`). Fields confirmed: `comment.id/.body/.user.login/.html_url`, `issue.number/.title/.body`, `repository.full_name`. Sources: https://docs.github.com/en/webhooks/webhook-events-and-payloads#issue_comment; https://docs.github.com/en/rest/issues/issues; https://github.com/octokit/webhooks/blob/main/payload-examples/api.github.com/issue_comment/created.payload.json
- H2. `issues` (action `opened`) is a **separate webhook subscription** from `issue_comment`; both require App permission **Issues: read** to subscribe. Posting a comment requires **Issues: write** (or Pull requests: write). Sources: https://docs.github.com/en/webhooks/webhook-events-and-payloads#issues; https://docs.github.com/en/rest/issues/comments#create-an-issue-comment; https://docs.github.com/en/apps/creating-github-apps/registering-a-github-app/choosing-permissions-for-a-github-app
- H3. Reactions: `POST /repos/{o}/{r}/issues/comments/{comment_id}/reactions` and `POST /repos/{o}/{r}/issues/{issue_number}/reactions`, `content: 'eyes'` among 8 values; requires **Issues: write** (no dedicated reactions permission). Octokit: `octokit.rest.reactions.createForIssueComment` / `createForIssue`. Source: https://docs.github.com/en/rest/reactions/reactions#create-reaction-for-an-issue-comment (re-verified)
- H4. Permission check: `GET /repos/{o}/{r}/collaborators/{username}/permission` returns legacy `permission: admin|write|read|none` where **maintain→write and triage→read** (doc-verbatim); fine-grained `role_name` + `user.permissions{pull,triage,push,maintain,admin}` on the same response. Callable with **Metadata: read**. Octokit: `octokit.rest.repos.getCollaboratorPermissionLevel`. Source: https://docs.github.com/en/rest/collaborators/collaborators#get-repository-permissions-for-a-user (re-verified) — the brief's "write/maintain/admin" gate = accept `permission ∈ {admin, write}`.
- H5. GitHub Apps are **not mentionable**: no autocomplete, no notification/webhook on `@app-slug` text — apps subscribe to comment events and pattern-match text themselves (community-confirmed; no docs.github.com statement exists). Bot comment author login is `app-slug[bot]`. Sources: https://github.com/orgs/community/discussions/53504; https://github.com/orgs/community/discussions/136436; https://github.com/orgs/community/discussions/110341
- H6. claude-code-action parity (source-verified): listens to `issue_comment created`, `issues opened/assigned/labeled`, `pull_request_review_comment`, `pull_request_review`; trigger phrase default `@claude`, matched case-insensitively with word boundaries `(^|\s)${escapeRegExp(triggerPhrase)}([\s.,!?;:]|$)`; permission gate via `getCollaboratorPermissionLevel` accepting only `admin|write`; non-User (bot) actors rejected by default. Sources: https://github.com/anthropics/claude-code-action/blob/main/docs/usage.md; …/src/github/validation/trigger.ts; …/src/github/validation/permissions.ts; …/src/github/validation/actor.ts
- H7. Rate limits: installation tokens 5,000 req/h baseline (scaling to a 12,500 cap); secondary limits ≤80 content-generating requests/min, ≤500/h — reactions+comments are content-generating; irrelevant at expected mention volume but caps any runaway loop. Source: https://docs.github.com/en/rest/using-the-rest-api/rate-limits-for-the-rest-api
