## Why

Archie auto-merges every linked PR the moment GitHub reports it approved and mergeable — hardcoded in the merge orchestrator (`checkAndMergeLinkedPRs()` / `triggerMergeCheck()` in `src/connectors/github/merge.ts`). There is no per-repo control: repos that need manual rollout, more eyes, or no bot merges at all get the same treatment as low-stakes ones. There is also no way for a human to say "merge it" and have that be the gate — the `merge_pull_request` tool merges immediately whenever GitHub state allows, with no approval step. Issue #139 asks for a single per-repo knob, default off, enforced in the engine rather than in prompts.

## What Changes

- Add an `autoMerge` boolean to repo-agent frontmatter (`metadata.archie.repos[].autoMerge`; the legacy singular `metadata.archie.repo.autoMerge` is auto-migrated like the rest of that shape). Default **false**. Threaded through `PluginRepoEntry` → `RepoEntry`/`AgentDef` — the two explicit field-copy points (`plugin-loader.ts`, `registry.ts`) that would otherwise drop it.
- Add a policy lookup, `isAutoMergeRepo(github)`, resolving via `findAgentDefsContainingRepo`: a repo auto-merges only when **every** agent declaring it sets `autoMerge: true`; a repo no registered agent declares (including dynamic, PM-spawned agents) is never auto-merged.
- Gate the merge orchestrator: ready PRs in non-auto repos land in a new "ready" bucket instead of being merged. The PM is reactivated **exactly once per ready state** to tell the Slack thread the PR is ready and can be merged on request — deduped via a persisted per-branch marker (`BranchState.merge_ready_notified`), so webhook bursts and restarts don't re-notify. `autoMerge: true` repos keep today's behavior: approved + mergeable → squash merge.
- Give `merge_pull_request` a policy check of its own: in an `autoMerge: true` repo it merges directly as today; in a non-auto repo it does **not** merge — it posts a merge-approval request (new `merge` approval type, mirroring the edit-mode flow: interactive Slack buttons, `suspendStatus()`, `deferTeardown()` pause) and the task waits for the user.
- Extend the approval machinery end-to-end for the `merge` type: `postInteractiveToUser` approvalType union, Slack `approve_merge`/`deny_merge` action handlers, the `POST /tasks/:id/approve` API branch, new `Task.handleMergeApproval`/`handleMergeDenial` (both take the expected PR identity from the resolving surface and verify it against the pending request atomically), and the debug MCP (`approve` tool enum + `wait_for_task` `APPROVAL_TYPE`). The requested PR is persisted in `TaskMetadata.pending_merge_approval` so resolution knows what to merge and survives restarts.
- On approval, the engine re-checks GitHub state and merges if mergeable, notifying the user via the PM; if not mergeable it reports why instead. The explicit path imposes **no Archie-side review-approval floor** — GitHub branch protection is the sole authority (a PR with zero reviews merges if GitHub says it's mergeable).
- Align the tool's mergeability condition with the orchestrator's via a shared helper (`clean` OR `blocked` + `mergeable=true`, the GitHub Rulesets workaround), removing today's divergence.
- **Deliberate behavior break:** on deploy, every existing repo stops auto-merging (no plugin frontmatter sets the flag). That is the point of the change; repo owners opt in later in archie-plugins.

## Capabilities

### New Capabilities
- `pr-merge-policy`: per-repo `autoMerge` frontmatter flag (default off), engine-enforced merge gating in both the orchestrator and the `merge_pull_request` tool, a once-per-ready-state "ready to merge" notification, and a `merge` approval type wired through Slack, API, Task resolution, and the debug MCP.

### Modified Capabilities
- `debug-mcp-task-waiting`: the "Block until a terminal or actionable state" requirement's approval-gate scenario enumerates the approval types (`edit_mode` or `research_budget`); the enumeration widens to include `merge` (delta in `specs/debug-mcp-task-waiting/spec.md`). Waiting/event-folding semantics are unchanged.
- `archie-e2e-harness`: the approval-gate requirement currently says the harness "SHALL approve" on `approval_requested`; it generalizes to resolving the gate with the decision the scenario's recipe prescribes — approve by default, deny where the recipe exercises a denial path (the new `merge-approval-deny` recipe) — delta in `specs/archie-e2e-harness/spec.md`.

- **Docs & prompts**: `docs/architecture/github-integration.md` (merge policy layer; also fixes the stale `repoInfo.pr_number` legacy-fallback code sample), `docs/architecture/edit-mode.md` (note the merge tool's own approval gate in non-auto repos), `prompts/repo-agent.md` + the `merge_pull_request` tool description (describe the policy-gated contract instead of "merges immediately"). archie-plugins repo (docs only, one separate PR there, coordinated with the engine deploy): `CLAUDE.md` frontmatter reference gains the `autoMerge` row, and the PM runtime skill `pm/skills/engineering-team/SKILL.md` is rewritten for the new default — it currently teaches "System auto-merges when approved + CI passes", which becomes false on deploy day.
- **APIs**: `POST /tasks/:id/approve` accepts `type: "merge"` (additive; merge-type requests must carry the pending PR's `github`/`pr_number`). `approval:requested` events may carry `approvalType: "merge"` — additive for SSE/JSONL/debug-MCP consumers.
- **Behavior**: repos without the flag stop auto-merging on deploy day (intended break). `merge_pull_request` in non-auto repos now pauses the task instead of merging. The tool also becomes slightly more permissive in auto repos (accepts `blocked` + `mergeable=true`, matching the orchestrator).
- **Sequencing**: open PR #169 (max-mode approval) touches the same approval surfaces (`tools.ts`, `task.ts`, `events.ts`, `routes.ts`, `plugin-loader.ts`, `registry.ts`, `types/task.ts`); PR #176 touches `tools.ts`. Changes here are additive (new union members, new fields, new handlers) to keep the conflict surface small; whichever lands second rebases.
- **Dependencies**: none new. No new GitHub App permissions (merge scope already in daily use).
