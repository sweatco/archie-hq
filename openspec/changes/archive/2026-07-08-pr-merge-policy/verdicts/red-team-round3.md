## NEW-1 (blocking): atomic resolution — DISCHARGED

- **Task-method-owned atomic gate.** D7 and tasks 3.3/3.4 give `handleMergeApproval(approver, expected: {github, pr_number})` / `handleMergeDenial(expected)` a **synchronous read-compare-clear** on `pending_merge_approval` — explicitly "no await between the three steps" in design (D7), tasks (3.3, 3.4), and the spec requirement text ("a synchronous read-compare-clear with no await between reading the pending request, comparing it to the expected PR, and clearing it"). Empty or mismatched slot → warn + stale disposition, slot untouched, no GitHub call. This is the exact fix round 2 prescribed.
- **Adapters parse-and-pass, no pre-verify.** Slack handlers (D6, task 6.1): ack → parse payload into `expected` → `Task.get` → `getUserInfo` → Task method; the handler "does **no** slot verification of its own", and both D6 and 6.1 record *why* (handler-side verify-then-await would reopen the race). The in-place message update moves **after** the Task method, driven by its returned disposition — a deliberate, documented ordering change from the edit-mode template. API route (task 6.2): `type: 'merge'` body must carry `github`/`pr_number`, missing → 400, passed through as `expected`. Debug MCP (task 7.1): `approve` gains `github`/`pr_number`, required when `type: 'merge'`, forwarded verbatim; the e2e recipe (9.1, AC3 row) supplies them.
- **The race test would actually catch the race.** Task 3.5: slot rewritten to PR#2 (simulated supersede) before `handleMergeApproval(approver, {PR#1})` → asserts **no merge of either PR**, **slot still holds PR#2**, **stale disposition returned**, and a follow-up `handleMergeApproval(approver, {PR#2})` still resolves normally; same mismatch case for `handleMergeDenial`. The round-2 failure mode was a slot-derived merge of PR#2 — the "no merge of either PR" assertion fails precisely under that regression, and "slot still holds PR#2" catches a mistaken clear. Mirrored in verification-plan AC4 row (d) and the 6.3 mismatch case. Beyond the test, the property is now structural: resolution merges only the PR named in `expected` — the PR whose prompt the user clicked — so no interleaving can merge a PR the user never approved.
- **Spec scenario covers mid-resolution supersede.** "Click on a superseded prompt is a no-op" now includes "when a supersede rewrites the pending request mid-resolution, after the click was received but before it resolves", and the requirement body states the pending request is cleared only on a matching resolution.
- **AC8 upgraded, not just repaired.** The identity check lives in the Task method, so "resolves identically" is now structural *including verification* (D6, AC8 row); the 6.3 test asserts both surfaces pass the same parsed `expected`.

## NEW-2 (non-blocking): task-level quiescence — DISCHARGED

- **Predicate is task-level and matches the prescribed source.** D6 step 2, D7 "Stuck-slot cancel path", and task 4.2 all gate suppression on `[...task.agentProcesses.values()].some((a) => a.pendingTeardown)` — verified identical to the `idleDecision` guard in `recovery.ts` (line ~101), with the rationale (slot is per-task, `pendingTeardown` is per-agent; a concurrent second repo agent must not supersede a seconds-old request) recorded in both design and tasks.
- **Concurrent-second-agent test exists.** Task 4.3: "slot set and a *different* agent process in the task holds the pending teardown → informational already-pending return, no supersede, no second prompt", alongside the stale-slot supersede cases (same PR and different PR).
- **Spec updated coherently.** "Duplicate request while the pause is parked is suppressed" explicitly includes the concurrently running second agent; the supersede scenario is renamed/reworded to "after quiescence (no agent in the task holds a parked pause)". The round-1 stuck-slot guarantee is preserved: `pendingTeardown` is consumed at turn end and defensively cleared at fresh run start, so after reactivation no agent holds it and supersede is reachable — the widened predicate cannot re-create a permanent block.

## Internal consistency and new-hole check

- D6/D7 ↔ tasks 3.3/3.4/4.2/6.1/6.2/6.3/7.1 ↔ spec ↔ verification-plan AC3/AC4/AC8 are mutually consistent on: signature shape, atomic gate location, adapter no-verify rule, message-update-after-disposition ordering, 400 contract, debug-MCP identity forwarding, and the race test's assertions. The migration plan's rollback remedy was updated to match the new API contract (deny body reads `github`/`pr_number` from the slot metadata).
- **No new spoofing surface from passing `expected` through adapters.** The Slack payload identity is engine-generated at post time and only *narrows* what a click can resolve — a forged/stale identity resolves nothing unless it matches the current slot; the trust boundary (who can emit a button click / call the API) is unchanged from edit mode. The API identity requirement adds no capability, only a constraint.
- **The 400 contract does not touch existing callers.** The current route validates `type`+`approve` generically and dispatches per-type (`routes.ts:230-268`); the merge identity requirement lives only in the new `else if (type === 'merge')` branch. `edit_mode`/`research_budget` API bodies, the existing debug-MCP `approve` usage, and the `edit-mode-approval` e2e recipe are byte-for-byte unaffected.
- Slot-cleared-before-merge-await (a supersede arriving during `getPRStatus`/`mergePullRequest` finds an empty slot and takes the normal fresh-request path, not the supersede path) introduces no conflict; slot-cleared-even-on-failed-merge is stated in D7 and asserted in 3.5.

## Remaining objections

1. **NON-BLOCKING — suppression predicate is not attributable to the merge park; condition both branches on the slot explicitly.** `pendingTeardown` is a single per-agent park shared by all approval types (edit mode, research budget, merge — cf. `onResearchBudgetExceeded`'s own `if (agent.pendingTeardown) return` at `task.ts:1182`). D6 step 2's first sentence infers "a merge request is already parked in this activation" from *any* pending teardown, which holds only when the branch is entered with `pending_merge_approval` set — and even then the park may belong to a different approval type. Two literal-reading edges: (a) if an implementer applies the teardown check before/without the slot check, a first-ever merge request issued while an unrelated park is live (e.g. another agent's research-budget prompt) returns a false "merge approval already pending" and posts nothing — the agent then believes a prompt is in front of the user that does not exist; (b) with a stale slot plus an unrelated park, suppression reports "already pending" for a prompt nobody can meaningfully answer — transient and self-healing (next reactivation supersedes), acceptable. Fix is one sentence in D6 step 2 / task 4.2: the suppression-vs-supersede fork applies **only when `pending_merge_approval` is set** (no slot → normal request path regardless of parks), which task 4.3's test premises ("slot set and …") already assume. No spec change needed — the spec's "a repeat call" wording already presupposes an existing pending request.

## Bottom line

NEW-1 is fixed the way round 2 demanded — verification moved inside the Task method, atomic with the clear, adapters reduced to parse-and-pass, and the race test's assertions genuinely discriminate the failure mode. NEW-2 is fixed with the exact prescribed predicate and test. The artifacts are internally consistent and the revision opened no new spoofing or compatibility hole; the one residual is a wording tightening on the suppression branch's slot precondition.
