## Verdict

**Gaps found — 2 blocking, 6 non-blocking.** The core engine design and its test plan are sound and verifiable; the blocking gaps are both cheap, well-bounded documentation/spec omissions that would surface as deploy-day PM misbehavior and a stale governing spec.

1. **blocking** — **Live PM skill still teaches auto-merge.** `/Users/khmelev/Projects/swc/archie-plugins/pm/skills/engineering-team/SKILL.md:28` says "**Merge** → System auto-merges when approved + CI passes" and line 78 says "auto-merge handles the green path". This is the PM's operative workflow skill (not the historical `prompts/_original/` copies). After the deliberate all-off break, the PM will promise users automatic merges that never happen and has no instruction about the new "ready — ask me to merge" flow. design.md D9 and tasks.md 8.3 update only archie-plugins `CLAUDE.md`; neither touches this skill. Fix: extend task 8.3 (same docs-only archie-plugins PR) to rewrite the two SKILL.md passages around the new default.
2. **blocking** — **Missing spec delta for `debug-mcp-task-waiting`.** proposal.md ("Modified Capabilities: None … requirements are approval-type-agnostic") is factually wrong: `openspec/specs/debug-mcp-task-waiting/spec.md:40` enumerates the approval type as "(`edit_mode` or `research_budget`)". Widening to `merge` without a delta leaves the archived main spec contradicting shipped behavior — exactly the "unverifiable claim later" failure. Fix: add `specs/debug-mcp-task-waiting/spec.md` with a MODIFIED "Block until a terminal or actionable state" requirement (one scenario wording change) and correct the proposal's Modified Capabilities section.
3. **non-blocking** — **e2e harness spec literalism vs the deny recipe.** `openspec/specs/archie-e2e-harness/spec.md:46`: "When a scenario's task reaches `approval_requested`, the harness SHALL approve it … observing the task proceed to completion." The new `merge-approval-deny` recipe (tasks 9.1) deliberately denies. Read literally, the recipe violates the SHALL. A small delta (or scoping the requirement to the edit-mode gate) keeps specs honest.
4. **non-blocking** — **No test for the merge API failing after approval.** design.md D7 promises the "merge-API failure message" lands in the decision finding, and `mergePullRequest` demonstrably returns `{success:false}` or throws (`merge.ts:174-187`). tasks.md 3.5 and verification-plan AC4 cover only not-mergeable *status* (dirty/closed), not a failed/throwing merge call. Add one case to `merge-approval.test.ts` (approve + clean + merge call fails → no completion finding, reason finding, PM reactivated).
5. **non-blocking** — **Ready-notification vs pending-approval interplay unaddressed.** design.md D4/D5 don't consult `pending_merge_approval` when notifying: if the approval landed while Archie was down (no webhook, marker unset) and the user requested the merge first, the next webhook-triggered check posts "ready — ask me to merge" while the approval prompt for the same PR is already pending — a confusing double prompt (no wrong merge). Suggest: skip notify (or set the marker) when the pending slot matches the PR.
6. **non-blocking** — **Stale Slack button can resolve a different PR.** Buttons carry only `taskId` (D6/D7); resolution reads whatever is currently in the slot. If a prompt is resolved via the API (the Slack message keeps live buttons — only the Slack-path handler updates it in place), a later request for a *different* PR refills the slot, and a click on the old message approves the new PR. Edit mode is immune (task-lifetime grant); merge is PR-specific. Cheap hardening: put `github#pr` in the button value and no-op on mismatch with the slot. Low likelihood, user-initiated, so non-blocking.
7. **non-blocking** — **Rollback with an in-flight merge approval strands the task.** design.md Migration Plan covers stale fields but not the pending prompt: after a rollback, old code 400s on `type: "merge"` and has no `approve_merge` handlers, so the paused task is unresolvable via the prompt and needs a thread nudge to reactivate the PM. One sentence in the Migration Plan (and operator awareness for AC9-day) closes this.
8. **non-blocking** — **AC1's "PM posts exactly one" has an unverified LLM half.** All pre-merge evidence is engine-side (one finding + one `sendMessage`); nothing observes the PM's actual single thread post (the AC3 recipe exercises the tool path, not the ready notification; AC9 exercises the merge click). Cheap closure: add "observe the ready notification appears once in the thread" to the AC9 operator checklist, and/or note the residual in verification-plan.md.
