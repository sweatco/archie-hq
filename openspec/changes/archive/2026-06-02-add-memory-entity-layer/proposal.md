## Why

The memory layer just reached a hardened v1, but its organizational knowledge is a single flat `org.md` of sectioned bullets. The durable **nouns** a multi-repo agent touches over and over — services, systems, integrations, recurring concepts — are buried inside those bullets, not addressable. That carries three costs:

1. **No targeting.** Every agent gets the *entire* `org.md` pushed at spawn regardless of what it's working on; relevance noise and prompt bloat grow with the file.
2. **Weak "related tasks."** Related-task linking uses lexical token-overlap of one-line summaries (`lifecycle.ts:441`), not shared subject matter. Two tasks that both touched `PaymentService` aren't recognized as related unless their summaries happen to share words.
3. **No canonical reference.** There is no single accumulating page for "what is `PaymentService` / `postgres-prod` / the Stripe integration," so the same facts get re-learned and re-stated across tasks.

Promoting these nouns to first-class **entity pages** with typed `[[wikilinks]]` turns the flat store into a navigable knowledge graph, and lets retrieval push only the entities a task actually touches — while staying file-based Markdown and push-only. The canonical spec already flags this moment ("Consider revisiting when org.md exceeds context budget or relevance noise rises").

## What Changes

- **ADD an entity layer.** A new `workdir/memory/entities/<slug>.md` per durable noun (`type: service | system | integration | concept | repo`). Each page has YAML frontmatter (`entity`, `type`, `display_name`, `aliases`, `scope`, `repos`, `domain`, `status`), an `<!-- L0: … -->` one-line summary, a typed-observations section using a **closed** category enum (`[fact] [config] [decision] [caveat]`), and a typed-relations section using a **closed** edge vocabulary (`depends_on · integrates · owned_by · part_of · touched_by · related_to`) over `[[wikilinks]]`.
- **People are not duplicated.** `users/<id>.md` remains the system of record for people; entity pages reference users by `[[<slackId>]]` wikilink.
- **ADD a derived entity index.** `workdir/memory/entities/index.md` is a thin table (one row per entity) **regenerated by housekeeping** — never hand-authored, never authoritative. Files win on conflict.
- **ADD entity-aware retrieval at spawn (push, not pull).** Always inject the thin index. Then deterministically select full entity pages by scoring the index against the spawn context (the agent's repo/plugin + participating users + task title), always include `scope: org` entities, and expand one hop along the wikilink graph. Inject a bounded top-N; **log what was dropped** (no silent truncation).
- **ADD entity extraction.** The extraction side-agent gains an `entity_updates` channel, and receives the current entity index so it **resolves against existing entities** instead of duplicating. Every write auto-adds a `touched_by [[<taskId>]]` relation.
- **MODIFY related-tasks selection.** Prefer entity-overlap ("tasks that share an entity") over the current lexical token-overlap; fall back to token-overlap when no entity signal exists.
- **EXTEND the trust boundary.** `sanitize.ts` / `paths.ts` gain hard **slug-as-filename** validation (regex, length, no path traversal) and a closed-vocabulary check for observation categories and relation types; an **entity-count soft cap** is added.
- **EXTEND housekeeping.** The consolidation side-agent gains entity **dedup/merge by alias**, stale-entity pruning (reusing `annotations.ts` `touched:`), and **index rebuild**.
- **REVERSE two Non-Goals.** The spec's *Domain-split files* and *Pull/query retrieval* non-goals are revised: domain becomes a first-class entity dimension (via frontmatter `domain`/`scope`, not directory splitting), and retrieval becomes *selective push* (still no agent-callable query tool).
- **RETIRE `org.md`.** Organizational knowledge stops being a flat file and becomes `scope: org` entity pages — always injected and now **exempt from the injection page bound** so org knowledge stays always-on regardless of entity count. Removed: the `org_updates` extractor channel, `applyOrgUpdates`/`readOrg`/`writeOrg`/`getOrgPath`, the `<organizational_knowledge>` injection block, `ARCHIE_MEMORY_ORG_CAP`, and org-file consolidation (`runHousekeeping('org')`). The single existing `org.md` fact is backfilled into a repo-scoped `backend` entity and the file is deleted. User-file housekeeping is unchanged — the *Org and user memory SHALL be housekept* requirement is replaced by a user-only *User memory SHALL be housekept*.
- **Future / research (explicitly out of scope here):** a **hybrid pull** model via a `read_memory` MCP/tool, **embedding/semantic** entity selection, and **full domain-directory** splitting. Captured as named follow-ups to try and measure, not built in this change.

This change retires the flat `org.md` store and its `org_updates` channel — organizational knowledge is not lost but reshaped into `scope: org` entities — and narrows the *Org and user memory SHALL be housekept* requirement to user files only. No other capability is retired.

## Capabilities

### New Capabilities

(none — this extends the existing memory layer)

### Modified Capabilities

- `memory-layer`: revise the **Non-Goals** (domain-split files; pull/query retrieval) and **ADD** requirements for: entity-page storage & schema; closed observation/relation vocabularies; derived entity index; entity-aware push selection at spawn; entity extraction with index-based resolution; slug-as-filename sanitization & entity soft cap; entity dedup/merge & index rebuild in housekeeping; entity-overlap related-tasks. **MODIFY** the existing *Memory injection at agent spawn* requirement to include the entity index + selected entity pages, and the *Per-task summary* requirement so related-tasks may be selected by shared entities.
- `memory-layer` (org.md retirement): **ADD** *Organizational knowledge SHALL be stored as entities, not a flat file* and *User memory SHALL be housekept*; **REMOVE** *Org and user memory SHALL be housekept*; further **MODIFY** *Memory injection at agent spawn* (drop `<organizational_knowledge>`; exempt `scope: org` from the inject bound), plus *File-based Markdown storage*, *Extraction triggers on task:completed*, *Prompt-injection defense in extractor*, and *Unmatched update actions SHALL NOT silently append* so none reference `org.md`.

## Impact

- **Spec**: `openspec/specs/memory-layer/spec.md` — Non-Goals + Glossary + Open Questions prose revised; 9 requirements added; 6 modified; 1 removed.
- **Architecture doc**: `docs/architecture/memory.md` — new storage-layout entries (`entities/`, `entities/index.md`), entity schema, retrieval-selection narrative, housekeeping entity rules, and a "Future directions" note for hybrid pull / embeddings.
- **Source** (`src/memory/`):
  - **New** `entities.ts` — read/write/parse entity pages (frontmatter, observations, relations), slug resolution against aliases.
  - **New** `entity-index.ts` — derive/rebuild `entities/index.md`; index-scoring + 1-hop graph expansion for selection.
  - `types.ts` — `EntityUpdate`, `EntityRecord`, `EntityRelation`, observation/relation enums.
  - `paths.ts` — `getEntityPath()`, `getEntityIndexPath()`, slug validator, entity soft-cap accessor.
  - `sanitize.ts` — slug validation, closed-vocabulary checks for categories/relations, path-traversal rejection.
  - `extractor.ts` + `prompts/memory-extractor.md` — `entity_updates` in the contract; index supplied as resolution context; the `org_updates` channel, `{{ORG_MEMORY}}` substitution, and the ORGANIZATION KNOWLEDGE prompt section are removed.
  - `context.ts` — inject entity index + selected entity pages into the memory block; drop the `<organizational_knowledge>` block; exempt `scope: org` pages from the inject bound.
  - `store.ts` — remove `readOrg`/`writeOrg`/`applyOrgUpdates`; `paths.ts` — remove `getOrgPath()` and `ARCHIE_MEMORY_ORG_CAP`.
  - `lifecycle.ts` — apply entity updates; auto `touched_by`; entity-overlap related-tasks.
  - `housekeeping.ts` + `prompts/memory-housekeeper.md` — entity dedup/merge + index rebuild.
  - `spawn.ts` — pass repo/plugin + task-title selectors into enrichment.
  - `index.ts` — bootstrap creates `entities/`.
- **Config**: `.env.example` — `ARCHIE_MEMORY_ENTITY_CAP`, `ARCHIE_MEMORY_ENTITY_INJECT_MAX` (bounded top-N).
- **Tests**: new cases for entity parse/write, slug validation, index rebuild, selection scoring + 1-hop expansion, entity extraction parsing, dedup/merge.
- **Ejectability preserved**: all new files live under `src/memory/` and `workdir/memory/entities/`; no new imports into core modules beyond the existing public surface.
- **Migration**: the entity layer is additive, but `org.md` is retired in the same change — its single existing bullet is backfilled into a repo-scoped `backend` entity and the file is deleted (n=1, a one-off move, not a general migration pass). Rollback remains `ARCHIE_MEMORY=false` or deleting `workdir/memory/`.
- **Risks**: slug-as-filename is the sharpest (model output becomes a path) → hard validation + soft cap; extractor cognitive load rises (entity resolution in `maxTurns: 1`) → may need a second pass, flagged in design; selection precision depends on repo/title signal quality → bounded top-N with logged drops and `scope: org` always-on.
