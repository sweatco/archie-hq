# Design: memory-v2-authz

Revised 2026-07-18 after the second adversarial review (max/strict): the `access: dm` cross-task read class is retired in favor of a DM write lockdown (prefs-only extraction). Sections 1–8 of tasks.md implemented the first iteration; section 9 tracks the rework to this design. Revised again 2026-07-18 after the third review (medium/strict, on these artifacts): classification errors get a distinct `unknown` class that read-locks (D1/D5), authorship body-framing moves in scope, and own-statement attribution becomes evidence-validated code-side (D3).

## Decisions

- **D1 — Fail-closed classification with a distinct error class, ext-shared wins unconditionally.** `getChannelVisibility` fails CLOSED to `unknown` — a first-class visibility value, not a collapse into `private`. `unknown` gates extraction like `private` AND triggers the full read lockdown like `ext-shared` (no injection, no tools, per-call deny): at error time the true class may be `ext-shared`, and `private` would grant the normal read surface — the third review's codex-1 hole (a transient API failure while a Slack Connect channel attaches would otherwise leak org memory into an external conversation). A missing stamp (pre-stamping records) stays `private` — write-gated but not read-locked: an unstamped legacy record predates stamping and restamps on its next message, whereas an error at attach time is precisely when a Connect channel might be arriving. `isChannelShared` keeps failing open (it only drives ephemeral warnings, where a missed warning is low-consequence and self-healing). Both ride one shared 60s `conversations.info` cache (errors not cached; the next call retries, so an `unknown` stamp clears on the next successful classification). `ext-shared` SHALL beat every other class with **no exceptions for D-prefixed ids**: Slack Connect DMs are `is_ext_shared` im conversations, so classification always consults `conversations.info` (the 60s cache bounds the cost) and evaluates the ext-shared predicate before the dm branch; `isChannelShared` loses its hardcoded `false` for D ids; `new_dm` channels are stamped via lookup, not a `'dm'` literal. The cost of failing closed is small by construction: `private` never blocks a task's own extraction restamp path — the next inbound message re-classifies.

- **D2 — Enforce at artifact creation, not only at read. Whitelist modes.** Read-side scoping alone cannot protect content extraction has already distilled into org-visible summaries, activity titles, and entity facts. The gate lives in `processExtraction` (after metadata load, before any store read) and returns one of three modes from the task's Slack channel visibilities, whitelist-style: any `ext-shared` → `skip`; else any `private` **or any value outside the known vocabulary (including missing)** → `skip`; else any `dm` → `prefs-only`; else → `full` (`access: org`). Only `public` and `dm` are recognized as non-gating — an unknown visibility string (hand-edited metadata, version skew) gates instead of silently classifying org. Skips emit one `extraction-skip` telemetry record; prefs-only runs emit `extraction-prefs-only`. Unstamped Slack channels gate as `private`: a task created before stamping shipped skips extraction until its channel restamps on traffic — accepted, measurable via skip telemetry. Gating also RETRACTS (review-3 claude-1): completion is not once-only — a reply reopens a parked task and a DM can attach mid-task, so a skip/prefs-only re-run for a task holding episodic artifacts from an earlier completion deletes the summary and drops the activity row inside the same serialized write pass; a stale `access: org` stamp must not keep fronting a log that absorbed confidential lines after it was written. Entity facts from the earlier run stay — distilled from the then-eligible transcript; per-fact provenance is a non-goal.

- **D3 — Authorship is the ownership boundary.** `extractAuthorUsers` scans only knowledge-log SOURCE lines (`[ts] [@<UID:Name> in …]`, tolerating the older no-channel-suffix shape) — body @-mentions never match, redacted external authors (`displayName === 'external'`) are excluded. This set is used everywhere a "task's users" notion previously meant mentions: extraction `allowedUserIds` (hard enforcement — updates keyed outside the set are dropped by the existing parser), extraction `<user_memory>` input, prompt injection, search user hits, and the pull-authz `authorUserIds`. Two hard-enforcement additions from the third review close the gaps the first iteration deferred:
  - **Body framing (was "out of scope", now in).** `formatLogEntry` frames message bodies at append time — continuation lines are indented — so no line originating from a body can match the source-line shape; `AUTHOR_LINE_RE` stays anchored to line start, and a crafted body mimicking `[ts] [@<UID:Name> …]` can no longer mint authors. Legacy unframed log lines keep the old trust class until their tasks age out (bounded residual, listed in Risks).
  - **Evidence-validated own-statements (was prompt-only).** Every `user_update` MUST cite ≥1 `msg:<ts>` source id as evidence; the lifecycle applies the update only when every cited id resolves to a transcript source line authored by the target user, and drops non-conforming updates with a telemetry record. The prompt's own-statements-only rule remains for quality, but the code-side check is the guarantee — a co-author's second-hand claim (or a transcript-injected instruction) can no longer reach another user's file via model misattribution. Framing (previous point) is what makes the msg-id → author mapping trustworthy.

- **D4 — Episodic read = `org` stamp or self. No dm class.** Decision order in `authorizeEpisodicRead`: (1) ext-shared caller denied (backstop; primary control is non-registration — checked FIRST so the ext-shared lockdown is unconditional, self reads included); (2) self allowed (agents already mount their own `shared/` dir read-only); (3) target stamped `access: org` → allowed for everyone; (4) everything else denied — no summary, no stamp, a legacy `access: dm` stamp, or an unknown value all parse to the same null and deny as `no-access-stamp`. `parseSummaryAccess` recognizes only `org` and harvests nothing else: `SummaryAccess` is the stamp alone; the channel-id/user-id frontmatter harvesting, `intersects`, and `MemoryToolsCtx.channelIds` from the first iteration are deleted. Rationale for retirement: the v1 dm grant (all task channels + all task authors) over-granted through mixed tasks' public parts — scoping it correctly (per-link visibility, DM-participant user lists) buys complexity for a class the owner decided should not be cross-task readable at all. DM tasks are episodically unreachable by construction (no artifacts exist); the read rule needs no dm case. `grep_task_log` reads the target's summary for the grant BEFORE `readKnowledgeLog` is ever called. Denials are normal results (not `isError`, avoiding agent retry loops), carry zero target content, and name the policy; task-ID existence is not treated as secret (activity rows already reveal IDs).

- **D5 — Ext-shared = total lockdown, enforced at spawn AND per call on all four tools.** No tools registered + no injection when any Slack channel has `visibility: 'ext-shared'`, `visibility: 'unknown'` (classification failure — D1), OR legacy `isShared === true` (any positive signal locks; the fail-closed stamp is the authority, the fail-open snapshot only ever adds). All three fold into the single ctx `extShared` flag — the per-call deny and telemetry reason stay `ext-shared`. Per-call `ctx.extShared` deny sits at the top of every tool handler — `search_memory`, `read_entity`, `read_task_summary`, `grep_task_log` — before any corpus read and before the self rule, covering stale spawn snapshots for the entity surface too (v1 covered only episodic reads; entity pages are distilled org knowledge — the exact material an external org must not see). Denials recorded in pull telemetry with reason `ext-shared`.

- **D6 — DM write lockdown: prefs-only extraction.** A `prefs-only` task runs the extractor with the same author-scoped `<user_memory>` input, then the lifecycle applies **only `user_updates`** (still restricted to `allowedUserIds`) and drops everything else the model returned: no summary write, no activity append, no entity updates, no Related Tasks participation. Enforcement is code-side in `processExtraction` — the prompt gains a mode hint ("this task is DM-derived; return user preference updates only") for token economy, but the lifecycle filter is the guarantee. Consequences: the activity index contains only org rows going forward (the v1 `Access` column + own-row injection carve-out are obsolete — injection filters to `access === 'org'`, legacy access-less and dm rows drop; the column is retained for parse tolerance of v1 dev stores); Related Tasks selection reads only org rows by construction, closing the v1 leak of DM titles into org summaries. DMs remain the richest preference source — preserving exactly that signal is the point of prefs-only over skip.

- **D7 — DM callers keep the full org read surface.** The lockdown is on what DM tasks *produce*, not what they *see*: a DM-task caller (extShared false) gets normal injection (entity index + pages, org activity, author prefs) and all four tools with the same rules as any org task. Explicit requirement + scenario so a later change doesn't "harden" DM callers into a lockdown the owner rejected.

- **D8 — Ejection story unchanged.** All authz logic lives in `src/memory/` (`authz.ts` is pure, no paths.ts import — tools tests keep their mock surface); core gains only the `ChannelVisibility` stamp (inert metadata) and the spawn-side ctx derivation on the existing seam. `MemoryToolsCtx` is exported through `src/memory/index.ts`; the two-seam-files invariant holds.

- **D9 — Cleanup postponed (owner decision), v1 artifacts included.** Legacy private/DM-derived summaries stay on disk but are unreachable (no `access:` stamp, or a v1 `access: dm` stamp → denied; no access column or `dm` class → dropped from injection and search). Entity pages may already hold private-derived facts — mitigations are the 6.1 human store review and the narrowed Non-Goal's evidence trigger for per-fact provenance. The flag never flipped in prod, so v1 dm-stamped artifacts exist in dev stores only; an optional one-off cleanup script (delete `access: dm` summaries + dm activity rows) is listed as a task, not a blocker.

## Risks / Trade-offs

- [Memory loss: private tasks and all pre-stamp tasks store nothing; DM tasks stop producing summaries/activity/entities] → deliberate policy; `extraction-skip` / `extraction-prefs-only` telemetry + eval report lines quantify the loss; active channels restamp on the next message so the pre-stamp effect decays quickly. DM episodic continuity (an agent re-reading a prior DM task's summary) is lost by design — the owner values non-leakage over continuity; user preferences carry the durable signal.
- [Authorship spoofing via crafted message bodies (D3)] → closed for new appends by body framing; legacy unframed log lines retain the old trust class until their tasks age out — bounded residual, and evidence validation independently blocks forged lines from driving user-file writes (a forged line's msg id either doesn't resolve or resolves to the real author's line).
- [Evidence validation may drop true positives (model cites the wrong line) (D3)] → dropped-update telemetry quantifies it; the extractor prompt requires citations, so a systematic miss shows up in the eval before the flag flips.
- [`unknown` stamps briefly lock down healthy tasks during Slack API outages (D1)] → deliberate direction of failure; the stamp clears on the next successful classification (per-message restamp, errors uncached), and the lockdown log line makes occurrences visible.
- [Visibility staleness: a public→private flip on a dormant task over-grants until restamp] → restamp on every inbound message + extraction re-runs on every completion, and a downgraded re-run now retracts the stale artifacts (D2) — the over-grant window closes at the next completion; the reverse direction (under-grant) is the failure mode by design.
- [`isShared` snapshot fail-open means a freshly-shared channel may briefly not lock down] → the stamped visibility (refreshed per message, fail-closed) is the primary trigger; the window is one message-processing cycle.
- [Classification now always consults `conversations.info` for D ids (D1)] → one extra API call per DM channel attach, amortized by the shared 60s cache; the alternative (API-free dm short-circuit) is exactly the Slack Connect DM bypass the second review confirmed.
- [CLI/self-launched tasks classify `org` with zero channels] → nothing human-confidential can have been ingested (no Slack append happened); the operator owns the store anyway.
