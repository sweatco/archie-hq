# Proposal: memory-v2-authz

## Why

An adversarial review (xhigh/strict, 2026-07-17) confirmed a HIGH finding against the pull path: `buildMemoryTools` used the spawn's taskId/agent for telemetry only — any agent on any task could `read_task_summary`/`grep_task_log` any well-formed task ID (raw `knowledge.log` transcripts included, DM and private-channel content and all), and `search_memory` ranked every user's preference file, returning the best-matching bullet verbatim. The spec's Non-Goal deferred access control "until a concrete leak risk is identified" — this is that risk, identified before the `ARCHIE_MEMORY_TOOLS` flip (rollout task 6.2 of `memory-v2-pull-and-eval`).

A second adversarial review (max/strict, 2026-07-18) of the first implementation confirmed the `access: dm` read class itself was the weak point: the persisted grant (all task channels + all task authors) over-granted DM content through its public parts, and DM activity titles leaked into org-readable summaries via Related Tasks. The owner's decision: retire cross-task DM readability instead of scoping it.

The owner's layer-wide policy, revised 2026-07-18: (1) private channels store nothing — neither user memory nor entities; (2) **DMs write user preferences only** — no task summary, no activity row, no entity updates, no episodic artifacts of any kind; DM content is never readable by any other task; (3) **DM tasks keep the full org read surface** — injection and all read tools work normally for a DM-task caller (the lockdown is on what DMs *produce*, not what they *see*); (4) user memory is strictly per-owner — only Bob's own authored statements may change Bob's file, and only tasks Bob authors in may read it; (5) tasks in externally-shared (Slack Connect) channels get no memory surface at all — nothing written, nothing injected, every tool call denied; (6) retroactive store cleanup is postponed — read-side fail-closed covers legacy artifacts, including `access: dm` summaries written by the first iteration of this change.

## What Changes

- **Channel visibility classification (foundation):** new `ChannelVisibility = public | private | dm | ext-shared | unknown` classified via `conversations.info` (new `getChannelVisibility`; on error it fails CLOSED to `unknown`, a distinct class that both gates extraction and triggers the ext-shared-grade read lockdown — collapsing errors into `private` would hand the full read surface to a channel whose true class may be `ext-shared`; deliberately opposite to the advisory `isChannelShared`, which stays fail-open for warnings; both share one 60s info cache, errors uncached so the stamp self-heals). `ext-shared` SHALL win over every other class **unconditionally — including `D`-prefixed conversations**: Slack Connect DMs are `is_ext_shared` im conversations, so the dm branch is evaluated only after the ext-shared check, and there is no API-free short-circuit for D ids. Stamped onto `SlackChannel.visibility` at every channel-attach/refresh site (inbound messages, `new_dm`/`new_thread` via lookup — no static `'dm'` literal, thread linking); missing stamp = `private` everywhere.
- **Extraction confidentiality gate (push), whitelist semantics:** classification of the completed task's Slack channels decides the extraction mode — any `ext-shared` → **skip** (nothing written); else any `unknown` → **skip** (reason `unknown`); else any `private` **or any unrecognized/missing visibility value** → **skip**; else any `dm` → **prefs-only**; else (all `public`, or no Slack channels — CLI/GitHub) → **full** with `access: org`. Only the two known-safe values (`public`, `dm`) ever contribute artifacts; unknown values gate — the gate can never fail open on vocabulary drift. Skips log + append `kind: "extraction-skip"` telemetry; prefs-only runs append `kind: "extraction-prefs-only"`. **Downgraded re-completions retract:** a reopened task that gained a DM/private/ext-shared channel re-completes into skip/prefs-only, which deletes the earlier completion's summary and activity row — a stale `access: org` stamp never fronts a knowledge log that has since absorbed confidential lines (telemetry notes `retracted: true`).
- **DM write lockdown (prefs-only mode):** a prefs-only extraction applies **user-preference updates only** (still author-scoped) and writes nothing else: no summary file, no activity row, no entity updates, no Related Tasks participation. Enforcement is code-side in the lifecycle — non-pref extractor output is dropped regardless of what the model returns; the extractor prompt additionally gets a mode hint to not produce org content (token economy, not enforcement). With no summary and no activity row, a DM task is episodically unreachable by construction — no read-side dm rule is needed at all.
- **`access: dm` retired:** extraction stamps only `access: org`. The episodic read rule collapses to: self → allow; ext-shared caller → deny; target stamped `org` → allow; everything else (no summary, no stamp, legacy `dm` stamp, unknown value) → deny. `parseSummaryAccess` recognizes only `org` (anything else parses to null ⇒ denied), and stops harvesting channel/user ids — `SummaryAccess` shrinks to the stamp alone, `MemoryToolsCtx` drops `channelIds` (author user ids remain for user-hit scoping and injection). Legacy `access: dm` summaries and dm-stamped activity rows from the first iteration are unreachable (denied / filtered); an optional one-off cleanup script may delete them (flag was never flipped in prod, dev stores only).
- **User-memory ownership:** the involved-user set switches from transcript @-mentions to AUTHOR users (source-line scan, redacted external authors excluded) — mention-only users are no longer writable by extraction, injected into prompts, or surfaced by `search_memory`. Two hard-enforcement additions from the third review: (a) **body framing** — `formatLogEntry` indents message-body continuation lines at append time, so a crafted multi-line body can never produce a line matching the source-line format and forge its way into the author set; (b) **evidence-validated updates** — every `user_update` must cite `msg:<ts>` source ids, and the lifecycle applies it only when every cited line was authored by the target user (own-statements is code-enforced; the prompt rule remains for quality, not as the guarantee). DMs remain the richest source of preferences, which is exactly what prefs-only mode preserves.
- **Pull-tool authorization:** `buildMemoryTools` ctx carries the caller's task id, agent, author user ids, and ext-shared flag (absent = fail-closed self-only). Episodic reads serve non-self targets only when stamped `access: org`. `grep_task_log` authorizes before the log is opened. `search_memory` pre-filters its corpus (user hits = caller's authors; summary/activity hits = org-stamped only; entity hits org-wide). Denials are normal, content-free, policy-worded results, recorded as `denied: true` + reason.
- **Ext-shared lockdown, enforced at every layer:** no tools registered + no injection at spawn when any channel is `ext-shared`, stamped `unknown` (classification failure), or legacy-`isShared`; AND a per-call `ctx.extShared` deny at the top of **all four** tool handlers (`search_memory`, `read_entity`, `read_task_summary`, `grep_task_log`) — evaluated before the self rule, so "any tool invocation from an ext-shared caller context SHALL be denied" is literally true, entity content included. All three triggers fold into the one ctx flag; denial reason stays `ext-shared`.
- **Inject-path consistency:** user-preference blocks author-scoped; `<recent_activity>` re-rendered from parsed rows filtered to `access: org` (dm/legacy rows never render — the own-row carve-out dies with dm rows). DM-task callers get the full injection surface like any org task.
- **Eval surface:** `memory:eval` telemetry aggregation gains denial counts/reasons (denied queries excluded from the store-gap list), extraction-skip counts, and prefs-only counts; one report line each.
- **Spec:** Non-Goal replaced with the enacted policy (remaining non-goals: retroactive cleanup, per-fact entity provenance, channel rosters); 4 ADDED + 4 MODIFIED requirements (below).

**Ordering constraint:** this change MUST land before `memory-v2-pull-and-eval` rollout task 6.2 (`ARCHIE_MEMORY_TOOLS=true`), and SHOULD be in effect for 6.1 (`ARCHIE_MEMORY_INJECT=true`) — the pre-authz inject path broadcast private-derived activity titles into every prompt. No new flags; everything rides the existing `ARCHIE_MEMORY*` gates. Note: the extraction gate becomes active wherever extraction already runs — that is the policy taking effect, not a regression; skipped and prefs-only tasks are visible in telemetry.

### Modified Capabilities

- `memory-layer`:
  - ADDED — Channel visibility classification and stamping (errors stamp `unknown` and read-lock; ext-shared wins unconditionally, Slack Connect DMs included).
  - ADDED — Extraction confidentiality gate (whitelist modes: skip / prefs-only / full; `access: org` stamping only; skip + prefs-only telemetry).
  - ADDED — User memory ownership (author-scoped writes, injection, and search hits; framed log bodies kill authorship forgery; evidence-validated own-statement updates).
  - ADDED — Ext-shared task memory lockdown (no tools, no injection, per-call deny on all four tools; `unknown`-stamped channels lock identically).
  - MODIFIED — Memory injection at agent spawn (author users; org-only activity re-render; lockdown carve-out; DM callers keep the full surface).
  - MODIFIED — Existing memory for ALL involved users SHALL be passed to extraction (involved = authors).
  - MODIFIED — Agent-callable memory read tools (org-or-self episodic rule; deny scenarios).
  - MODIFIED — Pull-call telemetry (denied records; extraction-skip and extraction-prefs-only records).
  - Non-Goal "Access control / channel visibility" replaced (policy enacted; narrowed residuals listed).
