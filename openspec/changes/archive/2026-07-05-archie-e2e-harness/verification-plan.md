# Verification plan — archie-e2e-harness

This run is self-hosting: at Stage 4 the built harness runs against its own branch, and the evidence it produces is the verification manifest for its own PR. Live evidence lands under `openspec/changes/archie-e2e-harness/qa-evidence/<AC-id>/` via the evidence writer's `--out-dir`; unit ACs cite test files and case names from the plain-CI suite.

The QA runner needs only: this plan, `brief.md`, a checkout of the branch with a valid `.env` (`ANTHROPIC_API_KEY`; no Slack tokens), docker, and the archie-debug MCP from `.mcp.json`. All harness invocations are documented in `.claude/skills/archie-e2e/SKILL.md`.

| AC | Criterion (abridged) | Method | Concrete scenario / check | Evidence location |
|----|----------------------|--------|---------------------------|-------------------|
| AC1 | Single documented invocation boots from a clean checkout and reports healthy; a broken branch fails clearly, not by hanging | live-e2e | **Healthy path:** run `npx tsx tools/e2e/boot.ts` on this branch; capture the success output (resolved base URL + `/health` body). **Broken path:** rerun boot with a deliberately broken `.env` (e.g. `ANTHROPIC_API_KEY` emptied in a scratch copy) or a tiny `--timeout-seconds` against a non-serving state; capture the non-zero exit and the diagnostics block (compose `ps` + log tail) and confirm the command returned rather than hung | `qa-evidence/AC1/boot-healthy.{json,md}`, `qa-evidence/AC1/boot-broken.{json,md}` |
| AC2 | A scenario run creates a nonce task via the debug MCP, waits, and returns terminal state + knowledge log | live-e2e | Run the **basic recipe** from SKILL.md: mint `E2E-<hex>` nonce → `create_task` with the nonce in the message → `wait_for_task(nonce)` looping on cursor while pending → record `TASK`, `STATE`, attribution, PM replies, and a `get_log` excerpt containing the nonce | `qa-evidence/AC2/scenario-basic.{json,md}` |
| AC3 | Edit-mode scenario detects `approval_requested`, approves via the API path, observes completion | live-e2e | Run the **edit-mode recipe**: change-request task against a repo the local plugin config declares → `wait_for_task` reports `approval_requested` / `APPROVAL_TYPE=edit_mode` → `approve(task_id, edit_mode, true)` (MCP tool over `POST /api/tasks/:id/approve`) → continued `wait_for_task` reaches `completed`; excerpt the `approval:requested`, `approval:resolved`, `task:resumed`, `task:completed` events from `get_events` | `qa-evidence/AC3/scenario-edit-mode.{json,md}` |
| AC4 | Each scenario writes an evidence file in the documented format; the file alone suffices for an independent reviewer | live-e2e | Two-part check: (1) the AC1–AC3 and AC6 evidence files above were each produced by `npx tsx tools/e2e/evidence.ts` and validate against `archie-e2e-evidence/v1` as documented in SKILL.md; (2) the Stage 4 **verdict reviewer** — who has not seen the diff — judges pass/fail for AC2 and AC3 from the evidence files alone and records whether they sufficed. A negative control (payload with a missing assertion) is rejected non-zero by the writer | `qa-evidence/AC4/format-check.{json,md}` + the reviewer's ruling in `verdicts/qa-reviewer-round1.md` |
| AC6 | After a run, no harness containers remain | live-e2e | After the AC2/AC3 session, run `npx tsx tools/e2e/teardown.ts`; capture its confirmation line and an independent `docker compose ps` showing no project containers | `qa-evidence/AC6/teardown.{json,md}` |
| AC7 | Non-trivial helper scripts are covered by tests runnable in plain CI (no docker) | unit | `npm test` (vitest, no docker) passes including `tools/e2e/config.test.ts` (URL/timeout precedence), `tools/e2e/boot.test.ts` (healthy / fail-fast on exit / cap / diagnostics, fake fetch+ps+clock), `tools/e2e/evidence.test.ts` (schema validation, inconsistent-result rejection, markdown render), `tools/e2e/teardown.test.ts` (compose-ps parsing: clean / survivor / malformed). CI run on the PR is the record | Test file + case names cited in `forge.yaml` AC entry; CI run URL on the PR |

Notes for the runner:

- Execute AC1's healthy boot first; AC2, AC3 in the same booted session (serial — the nonce window assumes one runner); AC6 immediately after; AC1's broken-path check either before the healthy boot or after teardown so it never races the live session.
- If the edit-mode gate never fires because no engineering repo is configured in the local workdir, report AC3 as BLOCKED with the prerequisite named — do not substitute a read-only task.
- Every live evidence file must be written through the evidence writer (not hand-authored) so AC4's format claim is exercised by the very files verifying the other ACs.
