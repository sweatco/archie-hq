# Design: memory-v2-pull-and-eval

## Context

The store has months of extracted content; injection is default-off in prod and turning it on needs a pre-enablement gate. Agents today see only what push selection injects at spawn — there is no way to reach the rest of the store mid-task, and the "pull recovers what push drops" claim in the roadmap is explicitly untested. On the measurement side, one sensor exists (the selection sensor, firing only when injection is on) and `pull-remote-data.sh` harvests the store and sessions — but nothing turns any of that into comparable numbers: no duplicate-rate trend for dedupe to be judged by, no regression benchmark for selector changes, no aggregation of the telemetry the sensors write, and no functional check that selection surfaces usable content. Meanwhile the memory-layer spec still narrates retired features (org.md backfill, the removed Slack post) and carries open questions that have been resolved for weeks.

This change is the roadmap's Phase 2: the pull path (read tools + pull sensor), the two-tier eval harness (mechanical + functional) that makes prod data QA-able on a schedule, the spec cleanup that gives future deltas an accurate baseline — and the **enablement gate** (D10), because the evidence the injection flip is conditioned on is a natural output of the eval, not a distinct tool.

## Goals / Non-Goals

**Goals:**

- Give every agent track a bounded, read-only pull path into the store (`search_memory`, `read_entity`, `read_task_summary`, `grep_task_log`), flag-gated and independently rollback-able.
- Instrument every pull call (the pull sensor) so pull behavior becomes measurable ground truth: push-recall misses, store gaps, actual usage rates.
- Stand up `npm run memory:eval` — mechanical (store-health, telemetry aggregation, selection regression) *and* functional (real implementation scored on a labeled question set) over pulled snapshots — so dedupe, budget tuning, and any selector change are judged by trend, benchmark, and end-to-end answerability, not anecdote.
- Produce the enablement-gate evidence (worst-case token bound + store-review reading list) and the snapshot series it and the trend metrics run on, so injection can be flipped on with evidence (D10).
- Clean the memory-layer spec of retired-feature residue.

**Non-Goals:**

- No runtime writes from agents — `remember`/`forget`, ADD/UPDATE/DELETE/NOOP, and contradiction-based invalidation stay in the write-path phase (roadmap Phase 3). The single-writer funnel (extraction side-agent) is untouched.
- No embeddings — search is lexical, deliberately (D3). The embedding decision is gated on this harness's own evidence (roadmap Phase 4).
- The functional eval (D5, D9) scores memory's ability to answer a *labeled question set* — a benchmark and a prod-transcript-derived set. It does **not** attempt live-task outcome attribution ("did memory change how real task X ended"); that needs months of records and stays in Phase 5. It is scoped to gate selection/injection/pull changes, not to compare answering models (synthetic evals rank retrieval changes reliably but rank answering-model architectures unreliably — negative Kendall-τ vs. human benchmarks in one study).
- No SDK-native memory-tool adoption in this change — the buy-vs-build spike moves to the phase whose problem it actually solves (D6).
- No change to push selection, injection bounds, or extraction behavior.

## Decisions

**D1 — Read tools are bespoke, in `src/memory/`, registered through one new seam.** The four tools live in `src/memory/tools.ts` and are exposed as one in-process MCP server (`createSdkMcpServer`, same pattern as the existing tool servers in `src/agents/tools.ts`), registered for every track when `ARCHIE_MEMORY_TOOLS=true`. This adds a deliberate third core→memory seam (today: `initMemory` in `src/index.ts`, `enrichPromptWithMemory` in `src/agents/spawn.ts`); `src/memory/CLAUDE.md` and the ejection recipe in `docs/architecture/memory.md` are updated in the same change — ejection gains exactly one removal step. Alternative — the SDK-native memory tool (now GA, `memory_20250818`; context editing separately remains beta) — rejected for this phase on two grounds: its six-command surface (`view`/`create`/`str_replace`/`insert`/`delete`/`rename`) is write-capable file CRUD with **no search or grep command**, so it would create a second, unsanitized writer bypassing the extraction funnel and the `sanitize.ts` trust boundary while still not providing the search surface pull actually needs; and third-party benchmarking (MemoryAgentBench, ICLR 2026) has vendor memory systems *underperforming* a same-backbone long-context baseline (Mem0 21.1%, Zep 24.0%, MemGPT 28.3% vs. 42.2%), so any buy decision must be made on our own harness's numbers, which don't exist until this change ships. Its real comparison target is the forgetting phase's write tools — the spike is re-homed there (D6). External validation for the bespoke shape: both frontier vendors converged on file-granularity memory access over a plain file store — Anthropic's GA tool is exactly file operations on a directory, and Letta v2 replaced its embedding-backed `archival_memory_search` tools with `open_file`/`grep_file`/`search_file` plus a background sleep-time consolidation agent, which is structurally our extraction side-agent + Markdown store.

**D2 — Every tool argument goes through the existing guards; results are size-bounded; no write surface exists.** `read_entity` validates via `isValidEntitySlug` and reads through the entity module; `read_task_summary`/`grep_task_log` validate task IDs with the same pattern the store uses for task dirs; nothing concatenates paths by hand (`paths.ts` invariant). Search results return identifiers + one-liners, not page bodies — the agent follows up with `read_entity`, keeping payloads small and making every read an explicit, recorded decision. `grep_task_log` returns at most N matching lines with line numbers. The server registers zero mutating tools, so "reads are safe to hand to all agents; writes stay funneled" is enforced by construction, not by prompt.

**D3 — `search_memory` is lexical and uses the selection scorer's tokenization.** Same `tokenize()` + overlap scoring family as `selectEntities`, extended over user files, task summaries, and recent-activity rows. Two reasons. First, the roadmap gates embeddings on evidence that selection precision/recall is the bottleneck *after* pull lands — shipping embeddings inside the pull path would destroy that experiment. Second, attribution: when the pull sensor shows an agent pulled a page push didn't inject, identical scoring means the miss indicts budgets or context signals, not a scorer discrepancy — the miss list stays actionable. Alternative — grep/substring search — rejected as the *ranking* mechanism (no ranking, and its misses would say nothing about the selector), but pattern-scoped reads stay available via `grep_task_log`. Lexical-first is also the evidence-backed default at this scale: three independent 2026 results have agent-driven grep/file-read over raw corpora matching or beating dense retrieval (DCI-Agent avg NDCG@10 68.5 vs 47.0 best sparse/dense baseline on BRIGHT/BEIR; grep beat dense vectors for every harness-model pair on a LongMemEval subset; a file-reading agent scored 72.5% vs 48.5% for the strongest RAG baseline on LongMemEval-V2) — all on corpora far *larger* than our ~300 pages, where embeddings have more, not less, to offer. Regime caveat honored: those are preprints with frontier-model backbones and partially reverse under different tool-result delivery, which is exactly why the embedding decision waits for our own pull telemetry rather than being settled by literature.

**D4 — Pull records share the telemetry file via a `kind` discriminator.** Pull records append to the task's existing `memory/tasks/<taskId>/telemetry.jsonl` with `kind: "pull"`; existing v1 selection lines have no `kind` and readers treat kind-less lines as selection records — no rewrite, no second file, `pull-remote-data.sh -m` keeps harvesting everything, ejection stays one `rm -rf`. Same fail-safe contract as the selection sensor: a sensor error logs a warning and never affects the tool result. Alternative — separate `pull-telemetry.jsonl` — rejected: two files to aggregate, same lifecycle, no isolation benefit.

**D5 — The eval has two tiers, and the functional tier ships now (reversing an earlier mechanical-only stance).** Tier 1, *mechanical* (no model calls): store-health distributions and the near-duplicate rate, telemetry aggregation, and selection-regression diffs — free, run on every snapshot, don't rot. Tier 2, *functional*: run the **real** memory implementation as system-under-test and score what it surfaces (full design in D9). The earlier plan deferred everything functional to Phase 5; that was wrong on the evidence and on the ask. Wrong on evidence: the field's central result is that mechanical/store-health numbers *do not predict* whether memory surfaces usable information — under a functional harness, commercial memory products scored *below a trivial BM25 baseline* (MemoryAgentBench Accurate Retrieval: Mem0 32.6%, Zep 37.5% vs BM25 60.5%) and long-context assistants dropped ~30% vs their own oracle-context ceiling (LongMemEval); a store can look healthy and still not answer. Wrong on ask: "does injection surface the right thing" is answerable now — we have a store, a deterministic selection path, and (this change) a pull path, which is all the system-under-test a functional harness needs. What genuinely stays in Phase 5 is the *live-task outcome* audit (did memory help/mislead/sit-unused across real completed tasks) — that needs months of accumulated pull/selection records and is a different question from "can memory answer a labeled question." The two research-grounded harness rules: evaluate through the production `selectEntities`/injection/pull code path with the agent harness held fixed (harness choice shifts accuracy as much as the retriever, ~16pp), never a re-implementation; and no external/vendor number substitutes for this harness (third-party evals contradict vendor self-evals wholesale).

**D6 — Selection regression replays *recorded* contexts, never reconstructs them; goldens are prod-derived and stay out of the repo.** A golden case is `{v, harvested_at, snapshot_date, ctx, expected: {selected, dropped}}`. The eval replays `ctx` through the production `selectEntities` against the snapshot and diffs against `expected`. Source of `ctx`: the selection sensor, which records real spawn contexts once injection is on — so the regression set is a **post-enablement** asset that starts empty and fills as the sensor runs. It deliberately does not reconstruct contexts from session artifacts (metadata archaeology, agent-config resolution): that reconstruction is fragile and unnecessary (D10). Pre-enablement selection quality is instead evidenced by the functional eval (D9) on a benchmark and a synthesized question set, which need no recorded contexts. Baseline semantics: goldens harvested from the same code+store must diff zero, so any selector change shows only *intentional* diffs, reviewed like a snapshot test. Goldens and reports live outside the repo and outside the runtime store (default `~/archie-snapshots/{golden,reports}/`, configurable): they embed prod task titles and Slack user IDs, which do not belong in git history — the repo commits only the format, harvester, and runner. The buy-vs-build spike from old Phase 2 moves to the write-path phase (roadmap Phase 3), where the native tool's write-CRUD actually maps onto the problem (see D1); by then the regression set exists to judge it with.

**D7 — QA-on-prod-data menu, and what this change picks.** A July 2026 adversarially-verified research sweep found **zero** surviving published claims on QA-ing agent memory against production data — nothing on telemetry replay, shadow eval of selection changes, store-review workflows, outcome attribution, or privacy handling for prod-to-dev store copies. There is no playbook to borrow; the menu below is first-principles, and that absence is itself the strongest argument for the sensors-now stance — ground truth can't be bought later. The full menu, for the roadmap: (a) snapshot-series trend metrics — this change (eval §store-health, over `snapshot-memory.sh`'s dated tarballs); (b) telemetry aggregation — this change (eval §telemetry); (c) selection-regression benchmark — this change (eval §regression, post-enablement); (d) human store review with a generated reading list — this change (D10, recurring as needed, and the pre-enablement content gate); (e) worst-case token bound from the real store — this change (D10, the pre-enablement cost gate); (f) live E2E memory scenarios on a booted branch instance (seed store → run task → assert injection/pull via the archie-debug MCP) — cheap extension of the existing `archie-e2e` harness, an option for verifying the tools change itself, not built here; (g) **functional eval — real implementation scored on a labeled question set, benchmark + prod-transcript-derived** (D5/D9) — this change, and the pre-enablement selection-quality gate; (h) LLM-judge *live-task outcome* audit (helped/misled/unused over real completed tasks) — stays in the value-eval phase, it needs accumulated records and is distinct from (g)'s question-answering; (i) on/off interleaving for outcome attribution — stays an open question at the roadmap level. This change picks (a)–(e)+(g); the one option deliberately *not* built is a reconstruction replay of historical spawn contexts from session artifacts — superseded by (g), which scores real selection against ground truth without the archaeology. (f) is the recommended QA for this change's own rollout.

**D8 — Rollout order: tools flip with or after injection, not long before.** With injection off, no `<entity_index>` is injected, so an agent has no catalogue of what's pullable — pull-only mode works (tool descriptions say what the store contains) but under-uses the store and would make early pull telemetry read as "agents don't pull". Recommendation recorded in the roadmap: flip `ARCHIE_MEMORY_TOOLS` together with or shortly after `ARCHIE_MEMORY_INJECT`, so the always-injected index provides the affordance and pull records measure genuine demand. Both flags stay independent one-line rollbacks. Affordance precedent: Anthropic's memory tool ships with an auto-injected system-prompt protocol telling the model to always check its memory directory before acting — the equivalent lever here is one line in the injected `## Organizational Memory` header pointing at the tools, which is the cheap first fix if pull rates come back near zero (risk 3).

**D9 — Functional harness: real implementation under test, surfaced-context + fixed-reader scoring, Oracle control, governed judge.** The functional tier wires our memory layer into the ingest-then-query pattern every published benchmark shares (LongMemEval submits `{question_id, hypothesis}` JSONL; LongMemEval-V2 needs only `Insert(trajectory)` + `Query(q)→context`; MemoryAgentBench feeds chunks then asks). Our mapping: **ingest** = run the extraction path over a question set's source transcripts into a throwaway store under `ARCHIE_WORKDIR`; **query** = run production `selectEntities` + injection render (+ optionally the pull tools driven by a small agent loop) to produce the surfaced context for each question. Then score two units:

- **Surfaced-context recall** — did the surfaced context contain the labeled evidence entities/pages? Computed without a model (evidence labels vs. selected slugs), a direct precision/recall on injection. This is the mechanical selection-regression (D6) generalized from "matches recorded expectation" to "contains the answer."
- **Answer correctness** — a **fixed reader model** answers from question + surfaced context only; a rubric **judge** scores against the gold answer. Three arms per question, reader and judge held fixed across all: *no-memory* (floor), *memory* (our surfaced context), *Oracle* (evidence-only context = reader ceiling). memory−no-memory is memory's value; Oracle−memory is recall loss vs. reader loss — the LongMemEval control that stops us blaming the selector for the reader's mistakes. Over-injection is measured functionally by re-scoring at tighter token budgets: if trimming the injected context doesn't move answer correctness, those tokens were waste.

**Judge governance** (from the LLM-as-judge literature, which is unambiguous that a naive judge misleads): the judge is a *rubric* judge (per-question-type, LongMemEval's own reached 97% human agreement); it is validated before it gates anything — chance-corrected **Cohen's κ** against a human-labeled sample, *not* raw agreement (raw overstates κ by 33.8–41.3pp), and measured **position bias < 0.10** even when test-retest looks perfect (a judge scored 0.99 test-retest with 0.19 position bias — stability is not validity); and it is a **different model family than the extraction side-agent**, because a writer and a same-family judge leak preference at ~28.7% and the leak is undetectable by inspection. The judge model, rubric, and its last κ/bias validation are stamped in the report header; an unvalidated judge runs but its numbers are marked non-gating.

**Ground truth, two sources.** (1) A portable public benchmark (LongMemEval-S, ~115k tokens, 500 Q) — a stable regression anchor and a sanity check that the harness itself is wired right, runnable before any prod data exists. (2) A **prod-derived set** synthesized from real task transcripts: an LLM generates QA pairs with evidence-entity labels, the store supplies distractors natively (the LongMemEval distractor recipe, but our own store is the distractor pool), and a human validates a sample — the label set, not the generator, is the trust anchor, which also sidesteps same-model write-and-grade self-confirmation. Prod questions and reports stay laptop-local, off-repo (they embed task titles and Slack IDs); the repo ships the harness, the synthesis subcommand, and the benchmark adapter, never the data. Cadence: mechanical tier every snapshot (cheap); functional tier on demand and before any selection/injection/pull change ships (LLM-in-the-loop cost and non-determinism make it a gate, not a per-commit CI step) — seeds pinned, reader/judge/harness identical across arms.

**D10 — The enablement gate is a set of eval outputs, not a bespoke tool.** Turning injection on in prod is conditioned on three pieces of evidence, all produced by the eval + snapshot tooling this change already builds, plus one human review:

- *Snapshots* → `scripts/snapshot-memory.sh` wraps `pull-remote-data.sh -m` into dated `~/archie-snapshots/archie-memory-YYYYMMDD.tgz`, on a laptop `launchd` schedule. Hard dependency of the eval's trend metrics (store-health deltas need a snapshot *series*) and the vehicle for watching the sensor after the flip. Gotchas the script encodes: schedule with `StartCalendarInterval` (a daily `Hour`/`Minute`), not `StartInterval` (whose firings during sleep are missed outright, while missed calendar firings coalesce on wake); pass the container name explicitly because the pull script's auto-detect uses `mapfile`, absent from the macOS `/bin/bash` 3.2 that launchd's default `PATH` resolves; append each run's outcome to a log so a failed scheduled pull is visible. No repo-committed plist (it embeds a user-specific SSH host); setup lives in the script header. Retention keep-all (memory-only tarballs are ~hundreds of KB at the 300-entity cap).
- *Store-review reading list* → the eval's `--report`, covering **every block the injection flag enables**: entities ordered by connectedness / observation count / page bytes and a staleness distribution, plus every `memory/users/*.md`, `recent-activity.md`, and the persisted `memory/entities/index.md` (injected verbatim as `<entity_index>`), each with size and the same suspicious-content flags (URLs, imperative override phrasing, base64-ish blobs). The script only sorts the pile; the ~1–2h review stays human. This is the one enablement risk no metric catches — a prompt-injection residue that would *run* in an agent — and one flag rollback un-injects future spawns, not a poisoned instruction that already executed.
- *Worst-case token bound* → computed by the eval from the real store, no per-spawn reconstruction needed: rendered index tokens + `ARCHIE_MEMORY_ORG_INJECT_MAX` × largest org page + `ARCHIE_MEMORY_ENTITY_INJECT_MAX` × largest non-org page (both after `touched_by` render truncation) + the summed rendered `<user_preferences>` blocks over every user file (a true bound — runtime caps neither involved users nor, therefore, that sum, but each user contributes at most one block) + rendered recent-activity, every term through the exported production render path (D3's rule) and tagged with the budget env values read back through the flag accessors. Pasted into the enablement PR.

*Not built:* a replay that reconstructs historical spawn contexts from session artifacts to preview per-spawn injections. Its purpose — "would injection pick sane pages" — is answered better by the functional eval (D9), which scores real selection against ground truth rather than eyeballing reconstructed injections, and continuously by the selection sensor the instant injection flips on; the residual "real recent contexts, not synthetic" angle is covered by enable-and-watch (budget ceilings bound blast radius in code, the sensor records every real spawn immediately, rollback is one flag). Reconstruction archaeology is fragile and not worth its marginal signal.

Enablement flip (human, gated): with snapshots running, the store review clean (junk/staleness/injection-residue cleaned up **on the prod store**, only while the app is stopped or verifiably idle — the memory layer serializes writes through an in-process queue that out-of-band edits bypass, with a rebuilt index after manual edits), the worst-case token bound acceptable, and the functional eval on the benchmark sane → flip `ARCHIE_MEMORY_INJECT=true` in its own PR carrying that evidence. Then D8's tools flip follows. The eval remains read-only over its snapshot throughout (verified by tree-listing + hashes before/after, catching even a stray `plugins-data/`).

## Risks / Trade-offs

- [Cross-task `grep_task_log` re-exposes unsanitized transcript content (the memory sanitizer never touched `knowledge.log`), widening the prompt-injection surface to other tasks' transcripts] → tool results wrap matches in an explicit untrusted-data framing block, are line-scoped and count-bounded, and carry no instruction to act; this is the same trust class as agents reading Slack content today. Accepted with bounds; revisit if the store review or telemetry surfaces abuse.
- [Pull round-trips add turns/latency vs. push's zero-cost injection] → search returns thin ranked hits (identifiers + one-liners), full pages only on explicit `read_entity`; the pull sensor measures actual per-task tool-call counts so the cost is observed, not assumed.
- [Agents may not use the tools, or use them badly (vague queries, no follow-up reads)] → that outcome is itself the measurement: pull-rate and zero-result telemetry either validate the pull hypothesis or kill it with data; prompt guidance (one line in the injected memory header pointing at the tools) is the cheap first lever before any heavier fix.
- [Near-duplicate metric (lexical) misses semantic duplicates, understating the problem dedupe must solve] → acceptable for a trend baseline: dedupe's success shows as the metric falling and as merged-page counts; the metric definition is versioned in the report so a later semantic upgrade doesn't silently rebase the trend.
- [Golden sets rot as the store moves (diffs become noise against a newer snapshot)] → each golden pins its `snapshot_date`; the runner warns when replaying against a mismatched snapshot; harvest is a one-command re-run, expected periodically.
- [Mixed-kind telemetry file breaks naive readers] → kind discriminator with kind-less-line fallback is specced; the eval's own reader is the reference implementation and skips unparseable/unknown-kind lines, counting them.
- [Third core→memory seam erodes the ejection story] → seam is documented in the invariants file and ejection recipe in the same change; still no DB, no external service, one flag removes the surface at runtime.
- [Functional eval is LLM-in-the-loop: cost, latency, non-determinism could make it flaky or unaffordable to run] → it's a gate, not per-commit CI (D9); seeds pinned and reader/judge/harness held fixed across arms; the mechanical tier remains the cheap every-snapshot signal, so the functional tier's cost is paid only when a selection/injection/pull change is up for a decision.
- [Prod-derived question set could self-confirm — same model family writes memory, generates questions, and grades] → the human-validated label sample is the trust anchor (not the generator), and the judge is a different family than the extractor (D9); the synthesis generator being a third relationship is noted in the report so a reviewer can weigh it.
- [Functional numbers could be over-trusted as absolute quality rather than relative gating signal] → report frames them as arm-relative (memory vs no-memory vs Oracle) and scoped to selection/injection/pull gating, never answering-model comparison (synthetic evals are unreliable for the latter); the benchmark anchor exists precisely to catch harness wiring drift that would make absolutes meaningless.
- [No pre-flip signal over *real* historical spawn contexts (D10 builds no reconstruction replay)] → accepted: budget ceilings bound blast radius in code, the selection sensor records real spawns the instant injection flips on, rollback is one flag, and the functional eval already proves the selection code surfaces answer-bearing pages on labeled data. Marginal value of eyeballing reconstructed injections over those is low; the archaeology's fragility is not worth it.
- [Bundling the enablement gate with the read-tools/eval build could delay the injection flip behind unrelated engineering] → the flip is a flag (ops), independent of code deploy: land the eval + snapshot code first and run the gate before the tools/sensor code is even exercised, so injection goes on as soon as the gate passes, without waiting on the pull path.

## Migration Plan

No runtime behavior change ships enabled: `ARCHIE_MEMORY_TOOLS` defaults off; with it unset, tool lists, prompts, and telemetry are byte-identical to today. Deploy order: land code → run the enablement gate (D10: snapshots + store review + worst-case token bound + functional eval on the benchmark) → flip `ARCHIE_MEMORY_INJECT=true` in its own PR carrying that evidence → flip `ARCHIE_MEMORY_TOOLS=true` with or after injection (D8) in its own PR carrying the first full eval report. Each flag is a one-line rollback, store untouched. Spec cleanup and roadmap revision carry no deploy at all.

## Open Questions

- Result-bound defaults: max search hits (propose 10), max grep matches (propose 50 lines), max tool-result bytes (propose ~8K chars per result) — tune after first telemetry.
- Should `search_memory` cover archived entities behind an explicit flag argument, or exclude them entirely until someone asks?
- Eval report retention/location naming (`~/archie-snapshots/reports/memory-eval-YYYYMMDD.md` proposed) — revisit if reports ever need to be shared beyond the operator's laptop.
- Functional fixed-reader and judge model choice: which model plays reader (cheap, fixed, ideally the prod agent's own tier for realism) and which family plays judge (must differ from the Sonnet extractor) — pin at build time, record in the report header; revisit if agreement/κ is weak.
- Prod question-set size for a stable signal: LongMemEval uses 500; how many human-validated prod questions are enough to gate a selection change here — start small (~30–50 validated) and grow, reporting confidence.
- Pull in the functional loop: score the *push-only* surfaced context first (deterministic, cheap), or also drive a pull-tool agent loop per question (realistic but adds nondeterminism and cost)? Propose push-only as the default arm, pull as an opt-in arm once push numbers exist.
