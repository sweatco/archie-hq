# Proposal: memory-v2-pull-and-eval

## Why

Push injection (Phase 1) is bounded and instrumented but still default-off in prod: turning it on needs a pre-enablement gate, and everything past it — budget tuning, dedupe value, embeddings, the forgetting model — is gated on measurements that do not exist yet. Push also only ever surfaces what `selectEntities` picks at spawn; agents have no way to reach the rest of a months-deep store. All three gaps close with what prod already produces: read tools give agents a pull path whose calls are *revealed ground truth* (a pulled-but-not-injected page is a measured push miss); an eval harness turns pulled snapshots and telemetry into recurring, comparable numbers *and* runs the real implementation over a benchmark to prove selection surfaces usable content — which is exactly the evidence the injection flag flip is gated on, so the same tooling produces the enablement gate. Separately, the memory-layer spec still narrates retired features (org.md backfill, the Slack "learned from this task" post, open questions long resolved) — future deltas would apply against a baseline that contradicts the code.

## What Changes

- **Memory read tools (pull path):** a new in-process MCP server exposing `search_memory`, `read_entity`, `read_task_summary`, and `grep_task_log` to all agent tracks (PM, repo, plugin). Read-only — writes stay funneled through the extraction side-agent. Gated by a new flag `ARCHIE_MEMORY_TOOLS` (default **off**, same opt-in discipline as `ARCHIE_MEMORY_INJECT`); independent of the injection flag, so pull can run with push on or off.
- **Pull sensor (permanent):** every read-tool call appends one typed JSON record (query/args, result slugs or hit counts, zero-result marker) to the task's existing `memory/tasks/<taskId>/telemetry.jsonl`, alongside selection records. This is what makes the eventual value eval honest — pull calls reveal store gaps and push-recall misses that no offline metric can.
- **Eval harness (`npm run memory:eval`), two tiers:**
  - *Mechanical* (no model calls): store-health metrics (growth vs. previous snapshot, near-duplicate rate, staleness distribution, per-page/observation distributions), selection-regression replay over a golden set of recorded selection contexts, and telemetry aggregation (injection rates, budget-drop frequency, rendered-token distribution; pull hit/zero-result rates once tools are live).
  - *Functional* (the tier the mechanical-only v0 was missing): runs the **real** memory implementation end-to-end as the system-under-test — ingest transcripts through the extraction path, then produce surfaced context through the production `selectEntities` + injection render + (optionally) the pull tools — and scores what it actually surfaces against a labeled question set. Two scored units on the standard pattern (LongMemEval / LongMemEval-V2 / MemoryAgentBench): **surfaced-context recall** against evidence labels (no model needed), and **answer correctness** by a *fixed reader model* over question + surfaced context, with an **Oracle** upper-bound arm (evidence-only context) separating retrieval failure from reader failure. Question sets come from two sources: a portable public benchmark (LongMemEval-style) as a stable regression anchor, and a **prod-derived set synthesized from real task transcripts** with evidence-entity labels (the "over my own data" half). Any LLM judge is governed (below), scoped to gate selection/injection/pull changes — not answering-model choice.
- **Judge governance:** where the functional tier uses an LLM judge, it SHALL be a rubric judge validated before it gates anything — chance-corrected agreement (Cohen's κ, not raw match) against a human-labeled sample, position bias below 0.10 — and from a **different model family than the extraction side-agent** (preference leakage between a writer and a same-family judge is measured at ~28.7% and is hard to detect). Report Markdown written outside the runtime store.
- **Enablement gate:** the eval also emits the evidence the `ARCHIE_MEMORY_INJECT=true` flip is gated on — a **worst-case token bound** computed from the real store (largest rendered org / non-org page after `touched_by` truncation × the budget ceilings + summed rendered `<user_preferences>` blocks + recent-activity + the always-injected index) and a **store-review reading list** (entities by connectedness / size / staleness plus every injected non-entity block — user files, recent-activity, the rendered index — each flagged for suspicious content: URLs, imperative override phrasing, base64-ish blobs) for the ~1–2h human content review. Plus `scripts/snapshot-memory.sh` (laptop `launchd`, dated memory-only tarballs) — the snapshot series the eval's trend metrics consume, and the vehicle for watching the sensor after the flip.
- **memory-layer spec cleanup:** retire migration-tense and dead-feature residue — org.md backfill scenario, `org_updates` narration, Slack post lines in Observability/Dependencies, stale task-summary glossary path, resolved Open Questions — and resolve the eval-artifact format question (eval tooling and telemetry are not the runtime store; Markdown-only applies to the store).
- **Roadmap restructure (companion, same branch):** `docs/proposals/memory-v2-roadmap.md` rebuilt around a two-track view (capability vs. measurement): this change is Phase 2; Phase 3 becomes one write-path rewrite subsuming dedupe *and* forgetting/modifying (same ADD/UPDATE/DELETE/NOOP mechanism) with an eval gate on forgetting; embeddings become a conditionally-triggered Phase 4; the end-state value eval (Phase 5) consumes the harness built here.

## Impact

- **New files:** `src/memory/tools.ts` (read-tool implementations + MCP server factory), `scripts/memory-eval.ts` (+ `memory:eval` npm script) with the functional harness (benchmark adapter, fixed-reader runner, judge caller), a prod-transcript question-synthesis subcommand, and the enablement-gate outputs (worst-case token bound + store-review reading list); `scripts/snapshot-memory.sh` (dated memory-only snapshots via `pull-remote-data.sh -m`, laptop `launchd`). Golden/question sets + reports + snapshots kept off-repo (laptop-local; they embed prod task titles and Slack IDs).
- **Modified:** `src/agents/tools.ts` (register the memory tool server per track — a deliberate third core→memory seam; ejection recipe grows one removal step), `src/memory/paths.ts` (flag accessor), `src/memory/context.ts` or a sibling for the shared telemetry appender + exported render helpers the eval's token bound reuses, `.env.example`, `docs/architecture/memory.md`, `src/memory/CLAUDE.md` (seam count; stale `org.md` mention), `openspec/specs/memory-layer/spec.md`, `docs/proposals/memory-v2-roadmap.md`.
- **No runtime behavior change with flags off:** `ARCHIE_MEMORY_TOOLS` unset → no tools registered, no telemetry, byte-identical prompts and tool lists. Injection stays default-off until the (human) flip this change now gates.
- **Prod rollout:** enablement gate first (snapshots + store review + worst-case token bound + functional eval on the benchmark → flip `ARCHIE_MEMORY_INJECT=true`), then flip `ARCHIE_MEMORY_TOOLS=true` with or after injection; each flag is its own one-line rollback, store untouched.
