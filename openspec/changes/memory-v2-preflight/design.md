# Design: memory-v2-preflight

## Context

Injection is default-off (`isInjectionEnabled`, `src/memory/paths.ts`) and the selection sensor writes only when injection is on (`recordSelection`, `src/memory/context.ts`) — so prod has a months-old store and zero selection telemetry. `selectEntities` (`src/memory/entity-index.ts`) is a pure function of `(records, ctx)`; both inputs are capturable offline: the store via `scripts/pull-remote-data.sh`, spawn contexts via session artifacts (`sessions/<taskId>/shared/metadata.json` for the title, `knowledge.log` for involved users) plus the same selector-shape logic `spawnAgent` uses (`src/agents/spawn.ts:530`). Replaying selection offline therefore runs prod's exact code path over an end-of-task approximation of each spawn's context, against a store snapshot that postdates the replayed tasks — a faithful forward-looking sanity check ("what would injection do if enabled now"), not a byte-exact reproduction of historical spawns (see Risks) — without touching prod.

## Goals / Non-Goals

**Goals:**

- Answer the enablement gate question with evidence: what would each recent spawn have injected, at what token cost, and does it look sane.
- Produce the worst-case token arithmetic for the enablement PR description from the real store, not estimates.
- Order the human store review (reading list by connectedness/size/staleness + suspicious-content hits).
- Start accumulating dated store snapshots on the laptop so later phases have store-at-time-T history.

**Non-Goals:**

- No prod-side app, runtime, or infrastructure changes. Store content is the one exception: cleaning up pages flagged by the pre-enablement review happens on the prod store (the repo ships pull-only tooling — no push mechanism — and cleaning only a local snapshot would leave prod injecting the flagged junk); that is a data edit to files the app owns, not a code or config change — and it runs only while the app is stopped or verifiably idle, because the memory layer serializes writes through an in-process queue (`src/memory/lifecycle.ts`) that out-of-band edits bypass (task 3.2).
- The replay script is disposable tooling: no test suite, no back-compat promises, expected to be deleted or rewritten once live telemetry supersedes it.

## Decisions

**D1 — Reconstruct contexts from session artifacts, not sensor records.** Sensor records don't exist pre-enablement (the sensor is gated on injection). The replay reads pulled `sessions/<taskId>/shared/metadata.json` (task title, spawned agents, attached repos) and `knowledge.log` (involved users), then builds `MemorySelectors` exactly as `spawn.ts:530` does. Alternative — enabling injection briefly in prod to collect records — rejected: that's the risk the pre-flight exists to avoid.

**D2 — Reuse the real modules against the snapshot via `ARCHIE_WORKDIR`.** `WORKDIR` binds at module load (`src/system/workdir.ts`), so the env var is the script's only entry point — `ARCHIE_WORKDIR=<extracted-snapshot> npx tsx scripts/memory-preflight.ts`, set before launch, no positional snapshot-path mode — and the script refuses to run unless the workdir contains `memory/` (and `sessions/` for replay), so it can never silently read the real local workdir. It imports `listEntities`, `selectEntities`, `buildMemoryContext`, and the render path so selection and token counts go through production code. Agent config is the one deliberate exception to module reuse: the production plugin loader is hard-bound to `<WORKDIR>/plugins` and creates `plugins-data/` as a side effect (`scanPlugins`, `src/system/plugin-loader.ts`), while the snapshot contains only `memory/` and `sessions/` — so the script takes a `--plugins-dir` argument pointing at a local `archie-plugins` checkout **at the tip of the prod-deployed plugins branch (`ARCHIE_PLUGINS_BRANCH`)** — an unpinned dev checkout can carry a different `metadata.archie.primary` or renamed agent and silently diverge the dominant `SCORE_REPO` signal — and parses agent frontmatter plus each plugin's `.claude-plugin/plugin.json` manifest in-script (blessed for disposable tooling; frontmatter/manifest parsing, unlike selection, has no silent-divergence stakes). The `plugin` selector is the manifest's `name` field — it appears nowhere in agent frontmatter — and agent ids map to files as `<key>-agent` ↔ `agents/<key>.md`. `plugins/` MUST NOT be placed inside the snapshot workdir to feed the loader — that would make the loader write into the snapshot, violating the read-only requirement. Alternative — reimplementing selection in the script — rejected: a reimplementation can silently diverge; byte-identity is the whole value.

**D3 — Export the render helpers instead of approximating.** `renderEntityBlock` and the `<user_preferences>` / `<recent_activity>` wrappers are private inline code in `context.ts`; export all three (e.g. `renderEntityBlocks(selection)`, `renderUserPreferencesBlock(ref, content)`, `renderRecentActivityBlock(content)`, with `buildMemoryContext` refactored to call the exported helpers) so every worst-case term in D7 — largest org / non-org page after `touched_by` truncation, the all-users sum, recent-activity — uses the real rendered bytes with the same `chars/4` rule the sensor uses; hand-copying any wrapper would violate the spec's "never a reimplementation" rule, and `buildMemoryContext` can't isolate the terms (calling it per-user perturbs entity selection via `SCORE_OWNER` and display-name token overlap). Per-spawn full-context estimates go through the already-exported `buildMemoryContext` (D7). Visibility-only refactor, no behavior delta, existing tests unaffected.

**D4 — Replay the spawns that actually happened; PM track is the floor.** Every task replays the PM spawn (no `repo`/`plugin` selector). For specialist spawns, enumerate every agent the task actually spawned — `metadata.json`'s `agent_sessions`, the only field synced from real spawns; `participants` also gains entries on owner assignment without any spawn (`createAssignTaskOwnerTool`, `src/agents/tools.ts`), so it may appear only as a labeled "assigned, possibly never spawned" supplementary row, never merged into the replay set — and resolve each one's `repo`/`plugin` selector by agent kind, matching where production reads it (`spawn.ts:530`). Statically-defined agents: agent config from `--plugins-dir` (D2) is authoritative — it is the only source of the scalar `repo.primary` (which `metadata.archie.primary` may point at a non-first repo, `src/system/plugin-loader.ts`) and of `plugin` selectors, which appear nowhere in session metadata. Dynamic agents: `dynamic_agents[].repos[0].github` (the primary, `src/agents/registry.ts`) from the snapshot's metadata — they never exist in config. When config is unavailable for a static agent, `metadata.repositories` is a usable fallback only if it lists exactly one repo for that agent; with more than one there is no primary marker, so emit an explicit ambiguous-primary partial-coverage marker instead of guessing entry `[0]` — a wrong primary silently misapplies `SCORE_REPO`, the dominant selection signal, which is exactly the divergence this decision exists to preclude. Genuinely unresolvable spawns are marked as partial coverage rather than guessed. Users come from the exported production functions — `readKnowledgeLog` (`src/tasks/persistence.ts`) + `extractUsernames` (`src/memory/lifecycle.ts`), the same pair `extractTaskUsernames` composes in `spawn.ts` — imported directly, never copied: a hand-copied mention parser can silently diverge, which is exactly the failure D2 rejects reimplementation to avoid.

**D5 — Reading list rides in the same script, and covers everything the flag enables.** A `--report` section lists entities ordered by relation count, observation count, and page bytes; staleness distribution; and grep hits for suspicious content (URLs, imperative override phrases, base64-ish blobs). It also lists the non-entity blocks the same flag injects — every `memory/users/*.md` file, `recent-activity.md`, and the persisted `memory/entities/index.md` (production injects its rendered rows verbatim as the always-on `<entity_index>` block), with sizes and the same suspicious-content grep applied — since user files inject on every spawn involving that user and recent-activity and the index on every spawn, a wider blast radius than any single entity page; a content review scoped to entities alone would let junk in those files pass the enablement gate unexamined. The human review stays human — the script only sorts the pile. Alternative — separate script — rejected: same inputs, one file keeps the tool disposable as a unit.

**D6 — Snapshots: thin wrapper + laptop launchd, keep everything.** `scripts/snapshot-memory.sh` wraps `pull-remote-data.sh -m -o ~/archie-snapshots/archie-memory-$(date +%Y%m%d).tgz` (dir and host configurable via env/args, skip if today's snapshot exists). The wrapper passes the container name explicitly — the pull script's auto-detect path uses `mapfile`, absent from the macOS `/bin/bash` 3.2 that launchd's default `PATH` (`/usr/bin:/bin`) resolves via the `env bash` shebang — and appends each run's outcome to `~/archie-snapshots/snapshot.log`, so a failing scheduled run is visible instead of silent; the header documents the plist `PATH` and SSH-key assumptions. Scheduling via a user `launchd` plist (`StartCalendarInterval` with a daily `Hour`/`Minute` — not `StartInterval`, whose firings during sleep are missed outright per launchd.plist(5), while missed calendar firings are coalesced into one event on wake); setup instructions live in the script header — no repo-committed plist, since it embeds a user-specific SSH host. Memory-only tarballs are ~hundreds of KB at the 300-entity cap, so retention is keep-all. Alternatives — docker-host cron (touches prod host) and GitHub Actions (prod SSH key becomes a repo secret) — rejected for footprint; gaps from a powered-off laptop are acceptable for a best-effort history.

**D7 — Worst-case arithmetic is computed, not asserted, and covers everything the flag enables.** `ARCHIE_MEMORY_INJECT` gates the whole read path, not just entity blocks: `buildMemoryContext` (`src/memory/context.ts`) also injects one `<user_preferences>` block per involved user and a `<recent_activity>` block, and the sensor's `renderedTokensEst` measures the entire joined context. The bound must cover the same surface, so the script prints: always-injected index tokens (rendered from the snapshot) + `ARCHIE_MEMORY_ORG_INJECT_MAX` × largest org page + `ARCHIE_MEMORY_ENTITY_INJECT_MAX` × largest non-org page (both after `touched_by` render truncation) + the summed rendered `<user_preferences>` blocks over every user file in the snapshot + rendered `recent-activity.md` tokens, alongside the observed per-spawn maximum. The user term is snapshot-derived, not sample-derived: runtime imposes no cap on involved users (`extractUsernames` is unbounded), so any multiplier observed across replayed spawns is not a bound — but a user contributes at most one block, so the all-users sum is. And it is rendered bytes, per D3's rule, not raw file bytes: production wraps each file in a `<user_preferences user_id=... display_name=...>` block, and skipping the wrapper would undercount. Per-spawn estimates likewise come from the exported `buildMemoryContext` over the reconstructed context — called without `taskId`, so `recordSelection`'s gate keeps the replay from writing telemetry into the snapshot — not from entity blocks alone; otherwise pre-flight numbers would systematically undercount post-enablement sensor readings. Budgets are an input, not a constant: the replay MUST run with the budget env values intended for prod and echo every env value that shapes rendered sizes — `ARCHIE_MEMORY_ORG_INJECT_MAX`, `ARCHIE_MEMORY_ENTITY_INJECT_MAX`, and `ARCHIE_MEMORY_TOUCHED_BY_INJECT_MAX` (a superset of the sensor's two-field `budgets` record, which omits the `touched_by` cap even though it truncates every entity block) — read back through the production accessors (`getOrgInjectMax()` / `getEntityInjectMax()` / `getTouchedByInjectMax()`), never raw `process.env`, so a misspelled variable surfaces as the default it actually produced instead of the value the operator thought they set. The report header also echoes the `--plugins-dir` checkout's HEAD SHA, branch, and dirty flag (D2): selectors come from that checkout, and evidence pasted into the enablement PR must show which config version produced it. This block is pasted into the enablement PR.

## Risks / Trade-offs

- [Old tasks' `metadata.json` shape drift → reconstruction fails for some tasks] → skip-and-count: the script prints coverage (`replayed N of M tasks; skipped: reasons`) so eyeballing is honest about its sample.
- [Spawn selector unresolvable (static agent absent from the `--plugins-dir` config — the only source of `plugin` selectors and multi-repo primaries — and snapshot metadata missing, ambiguous, or dynamic-agent entry absent)] → per-task "partial coverage" marker (including the explicit ambiguous-primary case) instead of silent wrong selectors; PM track depends on neither.
- [Reconstructed contexts are end-of-task, not spawn-time] → known skew, mostly conservative on token counts: title generation is fire-and-forget racing the first PM spawn (which typically runs with `taskTitle` undefined while the replay reads the final title), and `knowledge.log` grows over the task, so the replay sees a user superset of what early spawns saw. Optionally replay a PM-first-spawn variant with `taskTitle` unset to bracket the range.
- [Store snapshot postdates the replayed tasks] → entities written by a replayed task's own completion (or by later tasks) are visible to its replay even though production at spawn time could not have seen them; results answer "what would injection look like if enabled now for tasks shaped like these", not "what would task X have injected then". Optionally flag selected entities whose created/last-touched timestamps postdate the replayed task.
- [Laptop asleep → snapshot gaps] → mitigated by using `StartCalendarInterval` (D6): calendar firings missed during sleep are coalesced into one event on wake, per launchd.plist(5), so a closed lid delays the day's snapshot rather than skipping it; days the laptop is powered off entirely are still lost — accepted, snapshots are best-effort history, not an SLO.
- [Replay sample too thin to judge (few recent tasks with usable artifacts)] → surface the count; if coverage is poor, widen the window (`--limit`) or pull with sessions from a longer-lived host before judging.
- [chars/4 token estimate under/over-counts vs the real tokenizer] → same estimator the sensor uses, so pre-flight numbers stay comparable with post-enablement telemetry; absolute error is fine for a go/no-go bound.

## Migration Plan

None — no runtime change, nothing to deploy or roll back. The enablement flag flip happens later, by a human, in a separate PR whose description carries this script's arithmetic.

## Open Questions

- Replay window default: last 30 tasks (`--limit 30`) proposed; adjust after seeing how many tasks the snapshot actually contains.
- Snapshot cadence: daily proposed (tarballs are tiny); drop to weekly if the pull ever gets heavier.
