# Memory v2 — Roadmap

> **Status:** Phase 1 hardening shipped 2026-06-30/07-02 (`memory-v2-phase1`, `memory-v2-fixes`, `memory-v2-injection-bounds`) and the selection sensor 2026-07-08 (`memory-v2-selection-sensor`); the enablement pre-flight is in flight (`memory-v2-preflight`). Injection still default-off in prod. Eval re-scoped 2026-07-02: enablement is gated by a cheap pre-flight, and the full eval moved to the end where live usage data makes it honest (was: replay-based gate before enablement). Remaining phases below are proposals, not commitments — each needs its own OpenSpec change.
> **As-built:** [`docs/architecture/memory.md`](../architecture/memory.md) · **Spec:** [`openspec/specs/memory-layer/spec.md`](../../openspec/specs/memory-layer/spec.md)
> **Principle:** an eval is only as good as its ground truth, and our ground truth is generated by live usage — so early phases plant sensors (selection records, pull-tool records, store snapshots) and the eval phase at the end consumes them. A pre-enablement replay eval was considered and rejected: prod baseline is injection-off, so there is no incumbent recall to protect; replayed proxy labels (~1 month of `touched_by` edges) can't support the decisions it was meant to gate; and the recency-ossification loop it worried about (injection → behavior → extraction → re-stamp) is invisible to replay by construction.

## Phases

### Phase 1 — Stop the bleed, then turn injection on

Everything between "the store is a liability" and `ARCHIE_MEMORY_INJECT=true` running in prod. Not a metrics gate — structural bounds, permanent instrumentation, and an evidence-backed flag flip; budget defaults, the zero-signal recency floor, and any always-on org allowlist are tuned later from live sensor data (flag changes, no deploy).

Shipped:

- Hardening (`memory-v2-phase1`, `memory-v2-fixes`, `memory-v2-injection-bounds`): org injection bounded and relevance-gated (budgets are ceilings; zero-signal pages stay index-only), extraction no longer biased toward `org`, per-page observation cap at the persistence boundary, `touched_by` render cap. The always-injected thin entity index is the recall safety net.
- Selection sensor (`memory-v2-selection-sensor`, permanent): at spawn, one JSON record — selected slugs with scores and scope, dropped-over-budget, zero-signal count, rendered block tokens — appended to `memory/tasks/<taskId>/telemetry.jsonl`, so `pull-remote-data.sh` harvests it even in `-m` memory-only pulls and ejection stays one `rm -rf`.

Remaining (`memory-v2-preflight`, apply-ready):

- Pre-flight replay (disposable script, not maintained infra): pull prod data (`scripts/pull-remote-data.sh`), run recent tasks' reconstructed `SelectionContext`s through the real `selectEntities` against the pulled store, print would-be injections + rendered token counts over every block the flag enables; eyeball for nonsense (does a payments task pull `payment-service` or noise?). Worst-case token arithmetic goes in the enablement PR description.
- Store review (human, ~1–2h): skim everything injection would surface — entity pages, user files, `recent-activity.md`, the rendered index — for junk, staleness, and prompt-injection residue; content quality is the real enablement risk, and no selection metric measures it.
- Store snapshots: `scripts/snapshot-memory.sh` on a laptop launchd schedule, dated memory-only tarballs — store-at-time-T history for later evals, no prod-side change.
- Enable in prod (global flag flip, separate PR carrying the replay's arithmetic and the review outcome); watch the sensor for a few weeks; tune `ARCHIE_MEMORY_ORG_INJECT_MAX` / `ARCHIE_MEMORY_ENTITY_INJECT_MAX` by flag. Rollback is the flag.

### Phase 2 — Runtime read + buy-vs-build spike

- Spike the SDK-native memory tool + context editing behind a flag; decide buy-vs-build on the spike's own measurements (tokens, turns, latency, recall on replayed tasks) compared against live push telemetry from Phase 1.
- Read tools for agents: `search_memory`, `read_entity`, `read_task_summary`, `grep_task_log`. All agents read; writes stay funneled through the extractor.
- Pull sensor (permanent): record every read-tool call (query, results, whether a returned page was subsequently read) into the task's `memory/tasks/<taskId>/telemetry.jsonl` alongside selection records. Pull calls are revealed ground truth: a pulled-but-not-injected page is a measured push-recall miss; a zero-result search is a measured store gap. This is what makes Phase 5 honest.

### Phase 3 — Semantic dedupe (can run alongside Phase 2)

- Semantic `resolveEntity` + ADD/UPDATE/DELETE/NOOP write path (mem0-style).
- The only mechanism that actually shrinks the store — `ENTITY_CAP` has no teeth while housekeeping merges only alias-linked duplicates.
- The snapshot series from Phase 1 turns dedupe into a measurable trend (duplicate rate per snapshot) instead of a one-off cleanup claim.

### Phase 4 — Selection embeddings + runtime write/forget

- File-based embedding index (`entities/index.embeddings.json`, in-process cosine — no DB) for selection — only if Phase 5 shows push precision/recall is still the bottleneck after pull tools land, and any new selector must beat the current token-overlap one on the Phase 5 regression set before it ships.
- `remember` / `forget` tools behind the single write funnel; contradiction-based invalidation instead of date-only staleness.
- Kill or repurpose observation categories (they are consumed only as a render prefix and part of the dedupe/re-stamp key).

### Phase 5 — Memory value eval (on accumulated real data)

Runs on months of sensor data, answers "does memory earn its tokens", and leaves behind a regression harness. Every input is pullable from the live instance: selection records (Phase 1), pull-tool records (Phase 2), the store snapshot series (Phase 1), sessions (`metadata.json`, `events.jsonl`, `knowledge.log`), per-task summaries, and extraction applies (`touched_by` edges).

- Push value: how often injected entities are actually referenced in the task's knowledge log / output, and injected tokens per task. Injected-but-never-referenced pages are measured over-injection.
- Push recall, revealed: pages agents pulled that push didn't inject; user corrections that repeat facts already in the store (sampled from session events / Slack threads).
- Pull effectiveness: search hit rate; zero-result queries become the store-gap list that feeds extraction tuning.
- Store health trend across snapshots: growth, duplicate rate (Phase 3's success metric), injection concentration / page turnover — the recency-ossification loop is only observable here, on live data.
- Sampled outcome audit: LLM judge over ~20–30 real tasks — did memory help, mislead, or sit unused; every "misled" case gets root-caused.
- Deliverables: a periodic memory-health report (`npm run memory:eval`) and a golden regression set harvested from observed misses — the benchmark any future selector change (Phase 4) must beat.

> The 2026-05 eval harness (targeting the pre-entity `org_updates` write path, with no-ship defects flagged by its own adversarial review) was dropped from the stash 2026-07-08; its accumulation-test idea lives on as Phase 3's duplicate-rate trend.

## Open questions

- Who writes memory: specialists read, only extractor/PM write — or PM-only end to end?
- Forgetting model: date-based staleness (today) vs. contradiction-based invalidation.
- Index tiering: the always-injected index (~9K tokens at the 300-entity cap) becomes the dominant fixed cost once injection is on; Phase 5's reference-rate data decides whether it needs tiering.
- Outcome attribution: is Phase 5's sampled judge audit enough, or do we need on/off interleaving periods to attribute task outcomes to memory?
