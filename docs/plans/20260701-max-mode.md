# Plan — Issue #142: Per-task "max mode" upgrade (max effort by default, optional model swap), human-approved

## Context

Archie's coding (repo) agents run on a **fixed model + effort** — today `opus` at the default reasoning effort (`opus` is set in every repo agent's frontmatter in `archie-plugins`; `sonnet[1m]` is only the fallback for generic plugin agents). There is no way for a user or the PM to say "put more into *this* task." Issue #142 asks for a per-task upgrade, gated by **human approval** because the premium levers (a bigger model, or maximum effort) cost more.

The mechanism **mirrors the existing edit-mode approval gate** (the issue says so explicitly), applied to *model tier / effort* instead of *write access*. It's fully SDK-native — `model` and `effort` are already per-request options passed to `query()` — so no new infrastructure.

**Outcome:** a task-lifetime, human-approved "max mode" that, when active, upgrades each coding agent — **per-agent configurable from the `archie-plugins` frontmatter**, with a sensible built-in default. The built-in default for coding agents is **maximum reasoning effort** (a cost-conscious accuracy bump on the same model). A model swap (e.g. to **Fable**, `claude-fable-5`, available on the API today) is an explicit **per-agent frontmatter opt-in**, not the default — Fable is significantly pricier, so it is opted into per agent rather than applied to every coding agent automatically.

### Design decisions (locked with the user)
- **Model swap only** in this change. (The advisor tool is explicitly **out of scope** — see below.)
- **Per-agent frontmatter** controls the upgrade: `metadata.archie.maxMode: { model?, effort? }`.
- **Defaults when frontmatter is silent:** generic plugin agents **and** the PM → *unchanged* (max mode == normal). **Repo + dynamic agents** → "increase accuracy" = **maximum reasoning effort** (`effort: 'max'`); the model stays as configured (Opus).
- Frontmatter always overrides the default: any agent can opt in, **pin a premium model (e.g. `model: claude-fable-5`)**, and/or set a specific effort.
- The gate is **task-lifetime, one-way, human-approved**, and pauses the task pending approval — identical semantics to edit mode.

### Explicitly out of scope: the advisor tool
The advisor tool (`anthropic-beta: advisor-tool-2026-03-01`) is **not** implementable through the Claude Agent SDK's `query()`: the SDK's `betas` option is hard-coded to accept only `context-1m-2025-08-07`, and there is no way to inject a server-side tool definition or an arbitrary beta header (confirmed against SDK docs + open SDK issue #845). Enabling it would require turning the debug-only reverse proxy (`src/system/context-probe.ts`, which already points every agent's `ANTHROPIC_BASE_URL` at a localhost interceptor) into a production request-rewriter — real infra with multi-turn/`pause_turn`/streaming correctness work. It is a **separate, repo-agents-only follow-up**, captured in the new doc but not built here.

## Implementation

The whole feature is a mirror of edit mode. Reuse these existing pieces as templates:
- Request tool: `createRequestEditModeTool` (`src/agents/tools.ts:489`), registered in `createOrchestrationMcpServer` (`src/agents/tools.ts:1948`, PM-only).
- Approval handlers: `handleEditModeApproval` / `handleEditModeDenial` (`src/tasks/task.ts:1232` / `:1257`).
- Interactive post: `postInteractiveToUser` (`src/tasks/task.ts:585`).
- Slack actions: `approve_edit_mode` / `deny_edit_mode` (`src/connectors/slack/events.ts:217` / `:266`).
- CLI/API approval: `routes.ts:253` branch + `sendApproval` (`src/cli/api.ts:64`) + `TaskDetail.tsx` unions (`:120`, `:188`).
- Model resolution: `resolveAgentModel` (`src/agents/model-label.ts:27`), consumed at `spawn.ts:232` and in the footer `collectModelsUsed` (`src/tasks/task.ts:662`).
- Frontmatter parsing: `plugin-loader.ts:245-254` (effort/statusLabel/visibility under `metadata.archie`).

### 1. Types & frontmatter
- **`src/types/agent.ts`**: add `maxMode?: MaxModeSpec` to `AgentDef`, where `MaxModeSpec = { model?: string; effort?: 'low'|'medium'|'high'|'xhigh'|'max' }`.
- **`src/system/plugin-loader.ts`**: add `maxMode?` to `PluginAgentDef` (near `:108`); parse `metadata.archie.maxMode` next to `statusLabel`/`visibility` (`~:252`), validating `model` is a string and `effort` is in the allowed set.
- **`src/agents/registry.ts`**: carry `maxMode: agent.maxMode` into both `scanAgentDefs` branches (`:74` repo, `:102` plugin) **and** `buildPmDef` (`:368-386`, reads overlay `model`/`effort` today) so a `pm` overlay can also set `maxMode` — keeping the "any agent can opt in via frontmatter" claim true (PM default stays unchanged). Dynamic agents inherit the repo default automatically: `synthesizeDynamicAgentDef` (`:189-215`) sets `repo` + `model: 'opus'`, so `isRepoAgent` is true and they get the max-effort default (model stays Opus unless a spec sets `maxMode.model`) — no spec change needed.
- **`src/types/task.ts`**: add `max_mode?: boolean` to `TaskMetadata` (mirror `edit_allowed` at `:281`). **No approver field** — unlike `edit_approved_by` (consumed by `buildCommitAuthorEnv` for git authorship), nothing would read a `max_mode_approved_by`; the approver is captured in the `knowledge.log` `decision` finding instead. This also lets the Slack handler skip the `getUserInfo`/external-user dance (that skip only exists to keep external names out of git history, which doesn't apply to a model choice).

### 2. Model/effort resolution (`src/agents/model-label.ts`)
- `resolveAgentModel(def, maxMode = false)`: when `maxMode`, return `def.maxMode?.model ?? <normal>`; otherwise `<normal>` (`def.model || (isPm ? 'opus' : 'sonnet[1m]')`). **The model only changes when frontmatter sets `maxMode.model`** — there is no built-in model default, so a repo agent with no `maxMode` stays on Opus. Backward-compatible via the default arg.
- Add `resolveAgentEffort(def, maxMode = false)`: when `maxMode`, return `def.maxMode?.effort ?? (isRepoAgent(def) ? 'max' : def.effort)`; otherwise `def.effort`. This is where the **repo/dynamic "increase accuracy" default (max effort)** lives (import `isRepoAgent`).
- **No `MODEL_DISPLAY_NAMES` change needed** — `beautify()` already renders `claude-fable-5` → "Fable 5" (verified), for agents that opt into the Fable model via frontmatter. Use the full model id (the bare `fable` alias isn't an SDK-recognized model string).

### 3. Spawn wiring (`src/agents/spawn.ts`)
- At `:232`, compute `const maxMode = metadata.max_mode === true;` then `const model = resolveAgentModel(def, maxMode);` and `const effort = resolveAgentEffort(def, maxMode);`.
- Replace the effort passthrough at `:599` with the resolved value, **keeping the truthiness guard**: `...(effort ? { effort } : {})` (so an `undefined` effort isn't passed). Off-mode is byte-identical since `resolveAgentEffort(def, false) === def.effort`.
- Footer: thread `maxMode` into both `resolveAgentModel(...)` calls in `collectModelsUsed` (`src/tasks/task.ts:662`) so the grey footer reflects a frontmatter model swap (e.g. "Fable 5") when active. With the effort-only default the model is unchanged, so the footer is unchanged (effort isn't shown in the footer).

### 4. PM request tool (`src/agents/tools.ts`)
- Add `createRequestMaxModeTool(agent, task)` mirroring `createRequestEditModeTool`: idempotency (`task.metadata.max_mode === true` → no-op ok), pending-teardown guard, optional `channel` validation, `appendAgentFinding('Max mode requested: <reason>', 'decision')`, Block Kit `*Max mode request:* <reason>` with buttons `approve_max_mode` / `deny_max_mode` (value = taskId), `task.postInteractiveToUser(..., 'max_mode', channel)`, `task.suspendStatus()`, `agent.deferTeardown(() => task.stop())`.
- Register it in `createOrchestrationMcpServer` tool list (`:1952`). (PM-only, like edit mode.)

### 5. Approval handlers (`src/tasks/task.ts`)
- `handleMaxModeApproval(approverName?: string)`, in order:
  1. **Idempotency guard:** `if (this.metadata.max_mode === true) return;` (mirror the tool's guard at `tools.ts:506`; the Slack path is protected by the button-strip, but the API path isn't — without this a repeat approval would clear a freshly-spawned Fable session mid-work).
  2. **Clear the PM's pending teardown** (`this.agentProcesses.get('pm-agent')?.clearPendingTeardown()` — mirror `:1241`; load-bearing, else the deferred `task.stop()` fires and kills the task we just approved).
  3. `metadata.max_mode = true`.
  4. **Reset sessions for every agent whose model actually changes** (see D-gate). For each live handle in `this.agentProcesses` where `!isPmAgent(a.def) && resolveAgentModel(a.def, true) !== resolveAgentModel(a.def, false)`: null the **in-memory** id (`a.session.session_id = undefined`) **and** reset metadata (`this.metadata.agent_sessions[a.def.id] = { active: false }`) — both halves, mirroring the retry path at `spawn.ts:799-800`. **The effort-only default triggers no reset** (the model is unchanged) — only a frontmatter model swap does; a raised effort is a per-turn param the next turn picks up.
  5. `debouncedSave()`, `appendAgentFinding('Max mode approved by <name>', 'decision')` (name passed in from the surface; no metadata field), `sendMessage(AGENT_PROMPTS.existingTask, 'pm-agent')`.
- `handleMaxModeDenial()`: finding + re-engage PM (mirror `:1257`).
- Extend `postInteractiveToUser` `approvalType` union (`:585`) to `'edit_mode' | 'research_budget' | 'max_mode'`.

### 6. Approval surfaces (Slack + CLI/API)
- **`src/connectors/slack/events.ts`**: add `approve_max_mode` / `deny_max_mode` actions mirroring `:217`/`:266` — `ack()`, rewrite message to `✅/❌ *Max mode …* by <@user>`, call `task.handleMaxModeApproval(<user display name>)` / `handleMaxModeDenial()`. **Simpler than edit mode:** no `getUserInfo`/`isExternalUser` git-history dance — just pass the display name (or `<@userId>`) for the finding.
- **`src/connectors/api/routes.ts`**: add a `type === 'max_mode'` branch to the existing `if (type === 'edit_mode') … else if …` chain (`~:253`). Note `type` is declared `type: string` (`:222`) — there is **no** compile-time union here, only a runtime branch to extend.
- **CLI unions (widen together):** `sendApproval`'s `type` param (`src/cli/api.ts:66`) and the two `'edit_mode' | 'research_budget'` unions in `src/cli/components/TaskDetail.tsx` (`:120`, `:188`) must all gain `'max_mode'` in one go — `:376`/`:380` feed `focusedApproval.approvalType` straight into `sendApproval`, so a partial widen won't typecheck. No per-type rendering branch exists (the CLI renders from generic `event.data.text` at `:186`), so `max_mode` renders and `[y]/[n]`-approves with zero extra UI work.

### 7. PM prompt (`prompts/pm-agent.md`)
- Add `request_max_mode` to the **turn-ending** tools (`:33`) and the tool list (near `request_edit_mode`, `:164`).
- Add a short guidance block: when the user asks to "use Fable / activate max mode / use the best model," or for an unusually hard/high-stakes task, explain the cost trade-off via `post_to_user`, then call `request_max_mode(reason)`. It's a **task-lifetime grant** like edit mode — request once.
- **Also touch the workflow decision tree at `:343-357`** — this is where edit mode is walked step-by-step ("post_to_user → request_edit_mode → STOP") and is the load-bearing spot that actually makes the PM invoke the tool at the right moment. Add the max-mode analog here, not just a standalone blurb. Extend the two pre-flight checklist lines (`:227`, `:296`) too.

- **`archie-plugins/CLAUDE.md`**: add `metadata.archie.maxMode` to the frontmatter fields table — default behavior (repo/dynamic → max effort; others unchanged), and how to opt a specific agent into a model swap (`maxMode: { model: claude-fable-5 }`), with an example.

## Risks / assumptions to verify during implementation
- **Fable model id / context window** (only reached via a frontmatter `maxMode.model` opt-in — the built-in default is effort-only and never swaps the model): id is `claude-fable-5` (per the model-id reference + advisor compatibility table) — confirm the SDK accepts it as a plain `model` string. Repo agents rely on 1M context (opus is native-1M); if the large injected prompt overflows on Fable, the frontmatter can pin `claude-fable-5[1m]`. Flag both in the doc.
- **D-gate — session resume across a *model* change (only the frontmatter model-swap path; the effort-only default is exempt).** `buildQueryOptions` passes both `resume: sessionId` (`spawn.ts:596`) and a fresh `model` (`:567`) on every spawn. Flow for an agent with `maxMode.model` set: it investigates read-only → acquires a `session_id` → max mode approved → PM re-delegates → the agent would **resume its existing session** now that `resolveAgentModel` returns the new model. **Edit mode is NOT precedent** — it changes sandbox/clones/tools on resume but never the model, so nothing here proves a model swap survives `resume`. If the SDK pins the original model on a resumed session, the swap is a **silent no-op** and — worse — the footer (which shows the *intended* resolved model) would falsely display "Fable 5". **Decision:** `handleMaxModeApproval` resets the session of every agent whose resolved *model* changes (never the PM) before re-engaging. The reset **must null the in-memory `Agent.session.session_id`**, not just `metadata.agent_sessions`: `spawnAgent` reads the resume id from the live `Agent` (`existingSessionId = agent.session.session_id`, `spawn.ts:637`), and `save()` re-syncs metadata *from* the in-memory object (`task.ts:1088`), so a metadata-only clear is silently overwritten. Clear both halves, mirroring the retry path (`spawn.ts:799-800`). Context survives because specialists write findings to the shared `knowledge.log`, which a fresh spawn re-reads — making the switch correct-by-construction. **The effort-only default changes no model, so it triggers no reset**; a raised `effort` is a per-turn param picked up on the next turn (verify at runtime — low-risk, since the footer doesn't surface effort, a lag can't mislead). (`node_modules` isn't installed here, so SDK resume-vs-model behavior must be checked at runtime; if a resumed session is confirmed to honor a changed model, the reset can be dropped to also preserve in-session reasoning.)
- **No clone/sandbox implications** (unlike edit mode) — max mode is purely a model/effort choice, so `task.stop()` on pause needs no `cleanupClones` changes; the existing `edit_allowed` guard already preserves clones when both modes are active.
- `recovery.ts` needs no change — its pending-teardown handling is generic.

## Verification
1. `npm run typecheck` and `npm run build` clean.
2. `npm run test` — new `model-label` cases + full suite green.
3. **End-to-end (dev):** `npm run dev`, start a task, ask "use Fable / max mode"; confirm the PM explains then calls `request_max_mode`, the task pauses, and Slack (or the CLI `[y/n]`) shows the approve/deny buttons.
4. On **approve**: `metadata.json` gets `max_mode: true`; a `decision` finding ("Max mode approved by …") is in `knowledge.log`. **Default case:** repo agents now run at `effort: 'max'` (model unchanged → footer unchanged). **Frontmatter model-swap case** (an agent with `maxMode: { model: claude-fable-5 }`): that agent **respawns fresh** (not resumes) and its `system/init` log shows `Model: claude-fable-5` (the D-gate probe — meaningful only once the in-memory session is nulled per step 5.4); the footer shows "Fable 5".
5. On **deny**: task resumes on the normal model/effort, finding logged.
6. Confirm a **generic** plugin agent is unchanged under max mode; that the repo/dynamic default applies `effort: 'max'`; and that a frontmatter `maxMode: { model: claude-fable-5 }` override both swaps the model and forces a fresh session.
