# Memory v2 — Bounded Injection, Pull Tools, Public Store

> **Status:** Implemented in PR [#135](https://github.com/sweatco/archie-hq/pull/135) on `feature/memory-v2`. Prompt injection and read tools remain off by default.
> **As built:** [`docs/architecture/memory.md`](../architecture/memory.md) · **Forward plan:** [`docs/proposals/memory-v2-roadmap.md`](../proposals/memory-v2-roadmap.md)

This plan records the decisions that produced the current branch. Superseded intermediate designs are not part of the shipped contract; the architecture document is authoritative.

## Problem

Production telemetry measured a roughly 424K-token system prompt, including roughly 381K tokens of uncapped `scope: org` entity pages. Entity extraction favored org scope, org pages bypassed injection limits, and pages grew indefinitely.

The original memory layer also lacked a confidentiality boundary. It could extract cross-task memory from private conversations, and planned pull tools would have exposed raw task logs and arbitrary user profiles.

## Stage 1 — Bound push injection

- `selectEntities` uses independent `ARCHIE_MEMORY_ORG_INJECT_MAX` and `ARCHIE_MEMORY_ENTITY_INJECT_MAX` ceilings.
- Repo matches, participating-user ownership, task-title overlap, and one-hop relation expansion provide relevance signals. Zero-signal pages remain index-only.
- The thin entity index remains the discovery layer when full pages are omitted.
- Entity observations and rendered `touched_by` relations have separate caps.
- Extraction defaults to the narrowest applicable scope instead of `org`.

## Stage 2 — Instrument selection and consolidate task artifacts

- Each memory-enriched spawn records selected, dropped, and zero-signal entities plus budgets and a rendered-token estimate.
- Task summaries and telemetry live together under `memory/tasks/<taskId>/`.
- Startup migrates legacy flat summaries into the per-task layout.

## Stage 3 — Add the pull path

Three read-only tools are registered for every agent track when `ARCHIE_MEMORY_TOOLS=true`:

- `search_memory` performs bounded lexical search over active entities, public task summaries, recent public activity, and the current task authors' collaboration profiles.
- `read_entity` reads one entity page by slug or alias.
- `read_task_summary` reads one public task summary.

There is no raw-log tool and no mutating memory tool. Every call records pull telemetry when a task ID is available. The evaluation harness was split from this branch and now lives in stacked PR [#228](https://github.com/sweatco/archie-hq/pull/228).

## Stage 4 — Make task visibility the confidentiality boundary

The final design replaced the branch's intermediate channel-level ACL prototype with one immutable task field: `visibility: public | private`.

- Public Slack channels, including Slack Connect channels, and non-Slack automated tasks create public tasks.
- Slack DMs, private conversations, channel-info lookup failures, and legacy metadata create private tasks.
- Only public tasks run extraction and write collaboration profiles, entities, summaries, or recent activity.
- Private tasks can read the public store but never contribute to it.
- Stored summaries and activity need no access stamps because the store is public by construction.
- Raw `knowledge.log` files are extraction input, not cross-task memory.
- A task can bind only one Slack thread, so its visibility cannot change by attaching a conversation with a different confidentiality class.

External and guest Slack messages are filtered or redacted before extraction. Interactive authorization actions require a successfully classified internal actor.

## Stage 5 — Tighten collaboration profiles

- Profile files remain keyed by Slack user ID under `memory/users/`.
- Only actual message authors are writable and searchable for the current task; body mentions grant nothing.
- `cli:` and `local:` files remain legacy maintenance data but are not loaded or updated by extraction.
- Updates are limited to `Communication`, `Deliverables`, `Workflow`, `Decision Making`, and `Constraints`.
- Every update needs resolvable `msg:<ts>` evidence authored by the target user.
- Task summaries show only profile changes the store confirmed it applied.

## Deployment and rollout

The former channel-level policy may have written private-derived facts without recoverable provenance. Before deploying this design, snapshot and clear `workdir/memory/`; do not reuse the old store.

`ARCHIE_MEMORY_INJECT` and `ARCHIE_MEMORY_TOOLS` remain independently default-off. Their enablement evidence is produced by the separate evaluation change. Either can be disabled without stopping extraction.

Automatic housekeeping shares the runtime extraction queue. The manual housekeeping command runs in a separate process and must only be used while the Archie service is stopped.

## Verification

The PR verification contract is:

```bash
npm run typecheck
npx vitest run src/memory/__tests__/
```

Coverage includes visibility gating, Slack Connect public behavior, evidence-bound profile updates, entity persistence and selection, applied-only summaries, read-tool bounds, telemetry, activity, and crash recovery.
