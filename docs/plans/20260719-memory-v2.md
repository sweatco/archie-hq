# Memory v2 — Bounded Injection, Pull Tools + Eval Harness, Confidentiality Layer

> **Status:** Implemented (PR [#135](https://github.com/sweatco/archie-hq/pull/135), branch `feature/memory-v2`). Rollout gates still open — both read-path flags default off.
> **As-built reference:** [`docs/architecture/memory.md`](../architecture/memory.md) · **Forward plan:** [`docs/proposals/memory-v2-roadmap.md`](../proposals/memory-v2-roadmap.md)

This plan records what shipped in the five memory-v2 stages and the decisions behind them. It replaces the per-stage spec/proposal documents that previously lived under `openspec/`; the as-built behavior contract is `docs/architecture/memory.md`.

## Problem

Production telemetry measured a **~424K-token system prompt on every agent, every turn**, ~381K of it `scope: org` entity pages injected in full and uncapped. Four compounding amplifiers, all code-confirmed: the extractor prompt was biased toward `scope: org`; `pickScope` defaulted to `org`; `selectEntities` exempted org pages from every injection budget; and entity pages only ever grew (append-only observations, archive-only-when-fully-stale housekeeping). The reframe that drove the redesign: org knowledge is a *wiki* being treated like a *user-preference preamble* — inject the table of contents, let agents open articles on demand.

A second, independent problem surfaced by adversarial review during the work: the memory layer had **no confidentiality model**. Anything extraction stored was readable by every agent, the planned read tools would have served any task's raw transcript and any user's preference bullets to any caller, and Slack-Connect (externally shared) conversations got no special treatment.

## Stage 1 — Bounded, relevance-gated injection (`memory-v2-phase1`, `memory-v2-fixes`, `memory-v2-injection-bounds`, 2026-06-30 → 07-02)

- `selectEntities` applies two independent budgets — `ARCHIE_MEMORY_ORG_INJECT_MAX` (default 8) and `ARCHIE_MEMORY_ENTITY_INJECT_MAX` (default 8) — as **ceilings, not targets**: a page of any scope becomes an injection candidate only with a relevance signal (repo match, `owned_by` participant, token overlap, one-hop graph expansion), relevance-scored with last-touched recency as tiebreak. Zero-signal pages are never injected to fill spare budget and are not logged as drops.
- The thin `<entity_index>` is always injected in full — dropped pages stay discoverable, making the index the recall net under bounded injection.
- Per-page observation cap (`ARCHIE_MEMORY_ENTITY_OBS_CAP`, default 30) enforced at the persistence boundary (every write path, housekeeping merges included), newest-touched retained, re-affirmed observations refresh their `touched:` date instead of duplicating. `touched_by` relations truncated at render time only (`ARCHIE_MEMORY_TOUCHED_BY_INJECT_MAX`, default 10) — disk history intact for provenance and related-task selection.
- Extraction de-biased: scope defaults to the narrowest applicable, never `org` (prompt + `pickScope`).
- Data-loss fixes bundled en route: durable pending-extraction queue behavior, multi-user memory loading, unmatched-update no-op semantics.
- **Live-verified in Docker on a real store** (139 entities, 128 org): entity-injection tokens **~257K → ~41K (~84% cut)**; an A/B flag flip moved org injection 8↔128 on the running system.

## Stage 2 — Selection sensor + per-task layout (`memory-v2-selection-sensor`, 2026-07-08)

- One JSONL selection record per enriched spawn — what was injected, what was dropped over budget, zero-signal exclusions, budgets in effect, rendered-token estimate — appended fail-safe to `memory/tasks/<taskId>/telemetry.jsonl`. Telemetry cannot be backfilled, so the sensor ships before injection ever turns on.
- Per-task artifacts consolidated under `memory/tasks/<taskId>/` (summary + telemetry side by side; legacy flat summaries migrated at startup; single-step ejection preserved).

## Stage 3 — Pull path + two-tier eval (`memory-v2-pull-and-eval`, 2026-07-10)

- **Four read-only tools** behind `ARCHIE_MEMORY_TOOLS` (default off), registered per spawn as an in-process MCP server: `search_memory`, `read_entity`, `read_task_summary`, `grep_task_log`. Every identifier passes the `paths.ts` guards before filesystem access; results are size-bounded; zero mutating tools — writes stay funneled through the extraction side-agent.
- The SDK-native memory tool was evaluated and **rejected for this phase**: its six-command surface is write-capable file CRUD with no search/grep command (a second, unsanitized writer bypassing the extraction funnel), and third-party benchmarking (MemoryAgentBench) has vendor memory systems underperforming same-backbone long-context baselines — buy-vs-build is re-decided in the write-path phase, judged on our own harness.
- `search_memory` is deliberately **lexical, sharing the selection scorer's tokenizer**: a pulled-but-never-injected page then indicts push budgets or context signals, never scorer skew — and it preserves the roadmap's embeddings experiment (embeddings enter only if telemetry shows selection is still the bottleneck after pull).
- **Pull sensor:** every tool call leaves a `kind: "pull"` record next to the selection records (same file, kind discriminator, kind-less lines stay selection records). Pulled-but-never-injected = measured push-recall miss; zero-result search = measured store gap.
- **Two-tier eval** (`npm run memory:eval`, over snapshots from `scripts/snapshot-memory.sh`): mechanical tier (store health + versioned near-duplicate rate, telemetry aggregation, golden-set selection regression through the production `selectEntities`, worst-case injected-token bound through the production render path) and functional tier (real extraction/selection/injection as system-under-test; surfaced-context recall/precision against evidence labels; fixed-reader answer correctness with no-memory/memory/Oracle arms; LLM judge governed — Cohen's κ + position bias < 0.10 against a human-labeled sample, different model family than the extractor, stamped NON-GATING until validated). Goldens, question sets, and reports are prod-derived and stay out of git.
- **Enablement gate is a set of eval outputs, not a bespoke tool**: worst-case token bound + human store-review reading list + functional sanity pass; the injection flip ships as its own PR carrying that evidence.
- First prod run: worst-case bound **360,630 tokens** at default budgets → gate says no inject flip before store cleanup; push selection missed all 3 evidence pages on the probe question while **pull recovered 2/3** — the concrete selection-scoring improvement candidate.

## Stage 4 — Authorization / confidentiality layer (`memory-v2-authz`, 2026-07-19)

Closes the pre-authz hole before any flag flips; enforced at write AND read, everything fails closed.

- **Channel visibility foundation:** every Slack conversation classified `public | private | dm | ext-shared | unknown` via `conversations.info` — no API-free short-circuit for any id shape, because Slack Connect DMs are D-prefixed `is_ext_shared` conversations and ext-shared wins unconditionally. Classification errors stamp `unknown`, a first-class value that write-gates AND read-locks (the true class may be ext-shared); errors are never cached, so stamps self-heal. Stamped at every attach/refresh site; a missing stamp is `private` (write-gated, not read-locked).
- **Extraction gate, whitelist modes:** any ext-shared/unknown/private/unstamped/out-of-vocab channel → store **nothing** (+ `extraction-skip` telemetry); any DM → **prefs-only** (user-preference updates only — no summary, no activity row, no entities, no Related Tasks; enforced code-side regardless of extractor output); only all-public (or no Slack channels — CLI/GitHub tasks) → full extraction stamped `access: org`, the only class ever written. Downgraded re-completions **retract** the earlier summary and activity row — a reopened task that gains a DM must not leave a stale org grant over its grown log.
- **Read authorization:** episodic reads (`read_task_summary`, `grep_task_log`) serve non-self targets only with an `access: org` stamp — no summary, no stamp, a first-iteration `access: dm` stamp, or an unknown value all deny as `no-access-stamp`. `search_memory` pre-filters its corpus (user hits = the calling task's authors only; task/activity hits org-stamped only). Ext-shared/unknown callers are denied per call at the top of all four tools, entity content and self reads included, backstopping stale spawn snapshots. Denials are content-free policy results recorded in pull telemetry.
- **User-memory ownership, hard-enforced:** authorship (source-line scan) replaces mentions everywhere — extraction writability, `<user_preferences>` injection, search user hits, pull authz. Message bodies are framed (indented) at append time so a crafted body can't forge a source line; every user update must cite `msg:<ts>` evidence resolving to lines authored by its target user, else dropped with `user-update-dropped` telemetry.
- The first authz iteration's `access: dm` cross-task read class (channel/author overlap) was **retired** after a second adversarial review showed it over-granting through mixed tasks' public parts — DM content now contributes exactly its durable per-user signal (preferences) and nothing else; v1 `dm` stamps deny like legacy artifacts.
- DM-task callers keep the full org **read** surface (injection + all four tools) — the lockdown is on what DM tasks *produce*, not what they *see*. Retroactive cleanup of pre-policy artifacts is deliberately postponed: they are unreachable (no stamp ⇒ denied, dropped from injection), and the enablement store review covers entity pages.

## Verification

Typecheck clean; 793 tests across 48 files green (unit + integration across memory, tools, authz, slack classification, eval harness). Stages 1–2 live-verified in Docker (numbers above); stage 3 exercised against a prod snapshot; stage 4 shaped by three adversarial review rounds (the second retired the dm read class; the third added `unknown` fail-closed classification, body framing, and evidence-validated user updates).

## Rollout state (as of 2026-07-19)

Everything dark by default: `ARCHIE_MEMORY_INJECT` and `ARCHIE_MEMORY_TOOLS` both off. Flips are separate PRs gated on the enablement evidence; order is injection first (or together with tools — without the injected `<entity_index>` agents have no catalogue of what's pullable), authz shipped as the hard precondition for the tools flip. The concrete gate checklist and post-flip monitoring live in the [roadmap](../proposals/memory-v2-roadmap.md). Rollback is always the flag.
